On 9/24/06, Gary E. Miller <gem at rellim.com> wrote:
> > I'm just kinda thinking aloud. Do we rename all the commands
> > internally, and when we get a short command ('O'), call the long
> > handler ('@GPSDO') but set a compatibility output flag? Do it the
> > other way around?
>
> We could do something like the J command.  Use the old single letters
> by default.  Then step up to the full feature set on request.

I meant internally - I don't think duplicating code would be wise when
we could just call the existing command handler.

I was going to say that perhaps '@' by itself is special, causing gpsd
to escalate to v4, but then I remembered the 'L' command. Clients
wishing to use protocol 4 features must send 'L' first, to see if the
daemon support v4. From there, the v4 parser can be probed for which
commands it supports.

In this way, the gpsd will always be ready to handle any command it
can, and v4 clients can (and should) test to see what commands and
protocol version is available.
