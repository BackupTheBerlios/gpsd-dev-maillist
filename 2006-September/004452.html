ckuethe at BerliOS <ckuethe at mail.berlios.de>:
> Add the | command. This command allows gpsd to pass through a raw command
> from the client to the server; the reverse of super-raw (R=2) mode. This
> could conceivably be used to reflash your receiver's firmware, modify the
> port settings, tune the navigation engine to produce flawed results or any
> other thing one could do with a direct connection. This command is primarily
> for debugging purposes.

*Bad* idea.  Please back this change out. As the hacking guide says:

<h2>Allowing clients to ship arbitrary control strings to a GPS</h2>

Tempting -- it would allow us to do sirfmon-like things with the
daemon running -- but a bad idea.  It would make denial-of-service 
attacks on applications using the GPS far too easy.  For example,
suppose the control string were a baud-rate change?

The right way to support this capability would be to have such a pass-through
command interpreted by the logic that watches the control pipe.  That way only 
a user locally privileged as root can use it.  Otherwise we'd have to have some
sort of credentials and authentication system, which is way too complicated.

I've actually considered implementing that, but I didn't see any demand.
What's your actual use case?
