On 2/25/2011 1:05 PM, Greg Troxel wrote:
> C99 says that strcpy with overlapping strings leads to undefined
> behavior.  See 7.21.2.3 via
>    http://www.open-std.org/jtc1/sc22/wg14/www/standards
> at
>    http://www.open-std.org/jtc1/sc22/wg14/www/docs/n1124.pdf
>
> So code that relies on str*cpy doing anything in particular with
> overlapping strings is incorrect.

Well, one of the problems with using strlcpy at all is that it is 
non-standard.

"Other criticisms are that the functions are non-standard and that there 
are implementation differences between the BSD and Solaris 
implementations (the return value of strlcat, when there is no NUL in 
the destination buffer, differs)."
http://en.wikipedia.org/wiki/Strlcpy

But if it ever does find its way into a C standard, we can be sure it 
would behave the same as the other str*cpy functions concerning 
overlapping memory.
