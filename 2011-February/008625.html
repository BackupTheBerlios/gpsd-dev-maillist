On Fri, Feb 25, 2011 at 11:32, Michael Cook <mcook at bbn.com> wrote:
> Well, one of the problems with using strlcpy at all is that it is
> non-standard.

Yes, we acknowledge that it's nonstandard.

> "Other criticisms are that the functions are non-standard and that there are
> implementation differences between the BSD and Solaris implementations (the
> return value of strlcat, when there is no NUL in the destination buffer,
> differs)."
> http://en.wikipedia.org/wiki/Strlcpy

Rather than rolling our own, I included the one from OpenBSD (which
introduced strl*). I checked NetBSD, FreeBSD and Darwin this morning -
they also use this implementation.

> But if it ever does find its way into a C standard, we can be sure it would
> behave the same as the other str*cpy functions concerning overlapping
> memory.

>From time to time, BSD utilities add non-standard extensions, and
these extensions copy the GNU interfaces where possible... they got
there first, so they define the spec (until a standards body takes
over). In this case OpenBSD defines the standard behavior until
ISO/ANSI/etc. take over.
