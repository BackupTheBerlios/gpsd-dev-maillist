> First the documentation does not have any examples of the input protocol.
> 
> ?WATCH={"enable":true,"raw":1};

> this would have save me some time...

Weren't you looking at gpsd(8)?  It includes examples of all commands 
and responses.

> The SKY class returns a list of satellites these should be a class
> as well.  I recommend something like "class":"SATELLITE".

My rule on this is that subobjects are classes only if (a) code might
have to dispatch on the class type, or (b) it's necessary for
consistency with another context where the class has to be
self-describing (which is why the DEVICE objects inside DEVICES he a
class tag - it can also be tossed at the user outside of DEVICES when
the hardware status changes)

The satellite subobjects fail both tests - and the SKY response is quite 
huge enough as it is, thank you.

> Any reason for not using mixed case?  "Satellite" is so much easier on
> the eyes and in Perl all upper case classes are considered
> "reserved".  Currently, all of the gpsd classes are directly mapped
> in to the perl namespace Net::GPSD3::Return::*.

Yeah, guess why *I* used upper case -- same reason exactly.  Mixed
case is for private extensions to the object protocol.

> "TVP" is a locally used term.  We use PVT.  Let's use a term for the
>  class name like "FIX".  Technically, "TPV" is probably the most
>  accurate since you get time first, then position, then velocity.

gpsd uses "TPV" throughout.  I just checked.  I see no big advantage
in switch to "FIX", but I'm open to argument.

Would you be willing to join your Perl binding to this project?  
Seems like a reasonable thing for us to ship.
