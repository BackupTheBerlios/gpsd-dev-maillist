Lonnie Mendez <lmendez19 at austin.rr.com>:
>    I can imagine detection of gps devices using pl2303 being somewhat 
> difficult.  Do the known pl2303 gps devices all use vendor id 0x067b and 
> product id 0x2303 to associate themselves with this driver?

Yes.

>                                                            The rs232 db9 
> adapter I have with this chipset identifies itself by these values as well,

That's bad.
 
> so being able to access the product and/or manufacturer strings would come 
> in handy.  It might be necessary to use libusb to make an exact match in 
> this situation.  The bus path libusb provides could be matched with the bus 
> path proc/tty provides to sync each data source.  Thoughts anyone? 

I'm not sure how product and manufacturer strings would help.  Both the
Haicom and the BU-303 just ID as product "Prolific Technology, Inc." and
"PL2303 Serial Port"; I think that's the same info the hex givrs us.

The fundamental problem here is that the vendor/product ID pair only IDs
the USB-to-serial chipset, not the GPS behind it.  Your report turns 
this from a theoretical to an actual flaw, and pretty well sinks the idea
of scanning for USB hardware.

There is one thing that might save us: the USB class ID.  What is it on
your rs232 db9 adapter?
