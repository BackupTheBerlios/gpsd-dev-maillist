A few thoughts about this and the current build system...

The original design idea called for a 'lean, mean' gps machine.
I expected it to be run in a resource constrained environment.
I expected that to be some lttle embedded linux system, but it turns out that 
the handheld crowd had the same requirement.

So, keeping gpsd's footprint small is still of concern.

I don't think we should add convenience code to gpsd. I think that could be 
handled by another program. It just does all the h/w snooping and USB magic,
and drops a valid configuration for gpsd to pick up. (did I mention electra?)

Since it is not part of the gpsd binary, it could be written in Python, maybe 
with a nice gui frontend.

As for the current build system: blech...

I understand the uglyness comes from the fact that we use a shared library. Do 
we really need to do that? Or can we get away with a regular .a lib and link 
statically? Makes debugging so much nicer. I can see that glibc should be 
shared, but how many clients does our lib feed?
