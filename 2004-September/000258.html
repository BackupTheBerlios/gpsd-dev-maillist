I just ran through configure.in with a machete, chopping out tests that
looked unnecessary or obsolete.  It's maintainable now, I hope.

Since automake conditionalization now takes care of not building the
test clients when X is not present, there is no longer good reason for
the ad-hoc code insisting that X and Motif must be present.  This
change should make life a little easier for the embedded guys.

There is some possibility that these changes might break the build on
some weird old Unixes or under ancient versions of X.  My opinion is
that nowadays, having a configure.in that doesn't cause dizziness
and nausea is more important than backward compatibility to the Stone
Age (though this is not the choice I would have made five years ago).

But please test on your platforms.  If I've broken the build on any
reasonably current Linux or *BSD or Solaris we do want to fix that.

Oh, and a question:  does anyone know the good magic for defining feature
options?  I want to turn these

AC_DEFINE(FV18_ENABLE,      1, [San Jose Navigation FV-18 support])
AC_DEFINE(TRIPMATE_ENABLE,  1, [DeLorme TripMate support])
AC_DEFINE(EARTHMATE_ENABLE, 1, [DeLorme EarthMate Zodiac support])

into symbols that depend on --enable-FV18, --enable-Tripmate, and 
--enable-Earthmate, but autoheader barfs on my attempts with 
"missing template" errors.
