> 2) It might be possible for us to bypass the BXA limits, thereby could

I'm sure a binary editor would work just fine for fixing the ITAR
limits.  No need for the sources.

The altitude and velocity test is almost certainly done in software
after a position solution is determined.  The range-rate of the
satellites in orbit far exceeds the limits imposed by ITAR.  You can't
impose a hardware limit that prevents the gps hardware from, say,
doppler-tracking while on a fast-moving object without also preventing
it from tracking a satellite that is orbiting normally.  Ditto for the
altitude, if the front-end hardware could ever tell how far it is from
a satellite, it would certainly still need to track the satellite +/-
3,000km, so the ~18km limit of ITAR isn't going to be doable in the
front-end hardware either.  The only way to impose those low limits is
to do it after the position solution, which means the limits have to
be done in software. (Until the position solution stuff gets moved to
hardware that is.)

BTW.  That rule appears to only be an export restriction (like the
previous crypto farce.)  In theory I could mod my gps and use it for
weather balloon work (where one needs 130kft and mach 1.5 for the
initial descent).  As long as it didn't drift across any borders, in
theory, it'd be ok.
