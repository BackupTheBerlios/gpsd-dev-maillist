Eric,

GPS week value during EverMore GPS initialisation is important. Time is
synchronised correctly always but date needs "correct" GPS week. It is
suprisse for me I would expect that aboslute time information is transmited by
satelites.

Here are some data correction progress as observer from NMEA for some initial
GPS week values:
0x0000: 110216->090180->271185
0x0433: 120800->130800->160800->130705
0x0533: 090705->100705->130705

Regards,
Petr

> Petr Slansky <slansky at usa.net>:
> > It could be a result of initialisation in EverMore binary mode. Message
0x80
> > has some fields GPS week and GPS tow, I put 0 there. Do you think that
these
> > parameters are critical? Or are those values calculated from satelite
data?
> 
> They're used to set the internal GPS clock until it acquires a better
> time lock from the satellites -- I'm not quite clear on how and when
> the latter happens. 
> 
> Yes, I think zeroing these could have the effect you describe.  So I just
> added a unix_to_gpstime() function to gpsutils.c: call it with the 
> result of timestamp() as its first argument and use the results to fill
> in the fields in that message.
> 
> This will do the wrong thing if your system clock has been zeroed,
> but I don't see an alternative.  SiRF chips, and some others, have
> a flag mask in their binary init command to tell the GPS which fields
> are valid; too bad Evermore didn't do likewise.
