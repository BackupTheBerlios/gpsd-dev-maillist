> "Rob Janssen" <gpsd at xs4all.nl> writes:
> > Using bitfields to map out data that is externally defined is
> > implementation dependent.  You can get it to work fine, but in another
> > environment it may fail completely.
> 
> Yup.  Although the only back pressure we have on compiler writers is
> to start using them.  The alternative is much harder to maintain code
> that uses defines for byte-masks, defines for shifts and defines for
> byte-offsets from the beginning of some byte-array.  If you ever use
> the wrong mask with the right shift (or visa versa) or apply them to
> the byte at the wrong offset the compiler will be powerless to tell
> you.

My take:

Back in the days when most processors didn't have barrel shifters
and word-addressability was the rule, compilers were liable to add
padding to structs and bitfields, sometimes in unexpected places.
Bitfield layouts that didn't know about the compiler's padding
rules were a fast ticket to serious lossage.

But I think on modern byte-oriented machines, the risks associated with
using bitfields to map externally-defined data have dropped into the
acceptable range.  Especially if your compiler groks #pragma pack(1).
