Eric S. Raymond writes:
> If and when I put RTCM service in the daemon, I'll keep the first
> timeout (60 seconds without HELO means good-by Charlie) and treat the
> RTCM session like it's in GPS raw mode -- no timeout, but if the outbound
> buffers back up the client gets dropped.

I started out with a buffer fill causing a drop but I found that was
too aggressive.  Folks that used cell-phones to feed DGPS data to
their gps would often get dropped.  I finally added logic to time how
long the buffer had no reads on it.  Only when that passed some larger
timeout did I drop the connection.  Currently it is set at 180 seconds
but that number is probably still shorter than really needed.
