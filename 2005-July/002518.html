> > CPU this takes on an ARM, like on a PDA (the ARM has no floating point
> > hardware)?
> 
> This is from FastraX release notes:
> 
>        Optimized the CPU load. The CPU load is now down to approx. 70 % from
>         the previous 90 %.
>         This enables higher fix rate than before. Now the highest attainable
>         fix rate is 2 Hz, or 3 Hz if receiver is configured to use 8 channels.
> 
>         NOTE: Remember that correct 1 PPS functionality requires the use of
>               1Hz fix rate.

Sounds frightening, but there is no indication what CPU and clock were used
(many ARM microcontrollers have clock at 20 MHz or barely above--cheap and
energy efficient).

In the real-time field there are software that don't really compute the whole thing:
they use pre-computed table and then they interpolate. That's very fast, but not
practical for all tasks. Others use fixed point types (that's why Ada support these):
basically these are integers. A typical fixed point value is the type used by the
US Census Bureau for longitude & latitude: decimal degrees multiplied by
1000000 (i.e. a fixed precision, so the name). Obviously, the fixed precision
does not go well with every tasks.

RoadMap uses some of these tricks, due to its PDA-centric objectives. Using
floating point would have been faster than using integer on a high-end PowerPC
--so the approach is obviously not a MAC-centric one ;-).
