The Fastrax/iTrax chipset should be dead easy to support.  It won't
even need changes to the packet getter, as all the extension commands use
the $P syntax prescribed for vendor extensions and are perfectly
conformant NMEA.  Ghod, how I wish all chipset vendors had this much clue.

I don't see any problem in principle with supporting subsecond polling rates.

I'll work on a driver this weekend.  I might end up extending the gpsd
driver architecture a bit -- this is the first nontrivial extension
of NMEA I've seen and it suggests some possibilities.

Damn nice job of design on somebody's part, if I can believe the spec.
Unambiguous command-confirmation and error responses.  Room for 
extensibility.  Good documentation.  Be still, my beating heart...
