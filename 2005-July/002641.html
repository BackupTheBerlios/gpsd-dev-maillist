> Remco Treffkorn <remco at rvt.com> writes:
>> IIRC, bit fields are implemented as strings of bytes. As long as we
>> assume
>> ( don't get me started) a byte to be 8 bits, things are easy.

>> does not play into it. Think assembly.
>
> "A string of bits" hides the first problem.  Which end does the string
> get packed from?  Does the first bitfield go in the lowest or highest
> numbered position?  The answer ends up depending on endianess.

Isn't it up to the compiler writer how bitfields are implemented?
It could be different with different compilers on the same architecture.

Same as with padding: a compiler writer can chose to pack all members as
tightly as possible into a structure, or individual members could be
aligned on convenient boundaries to allow efficient access.  Often there
are compiler options and/or pragmas to select this.

Using bitfields to map out data that is externally defined is
implementation dependent.  You can get it to work fine, but in another
environment it may fail completely.
