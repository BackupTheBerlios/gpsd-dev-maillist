Pascal F. Martin <pascal.martin at cox.net>:
>                 I tend to like callbacks: when certains documented
> conditions are met, call an application provided function (for
> example when a new position has been received that is not the same
> as before :-).

I considered that approach and rejected it because I didn't want the client
library to take over the application control lflow,

> This being said, that cannot be me: I am somewhat afraid of a program that
> scans all "unused" serial ports when it starts.

That's not how it works.  I managed to be smarter than that :-)

gpsd does no device scanning at all.  Instead, it relies on a hotplug
script that tells it when a device that might be a GPS has gone active.
When that happens, it opens the device and watches the output.  It then
autobauds and watches for well-formed NMEA or vendor-binary packets. 
Only once it has seen a well-formed packet does it send any probes
or control strings.

> What I do like with the gpsd protocol is that its all "named
> parameters" instead of "position defined parameters" like NMEA. That
> makes compatibility check a hell lot easier.

gpsd now has an "l" command that returns the protocol version and a
list of available commands.
