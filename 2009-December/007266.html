> >  Well, the logs show a difference, syncing directly to the PPS source

> > at 1000 ppm, it's useful.

> Not sure what you are saying here.  We always need the NMEA to get the
> PPS to work.  And I am not worried about the code cost, but the chance
> of getting bad fixes.

Since you have put back in the code that references PPS to GPS messages
instead of local time, ntpd/chrony don't need NMEA timestamps at all.
No matter how wrong the local clock is, gpsd will report correct
offset for PPS and ntpd/chrony can sync directly to it.

For users that need to have system clock synced as fast as possible,
this gives better results than the older gpsd approach or using system
PPS support (LinuxPPS) which need to be referenced to another time
first.
