On Tue, 18 Dec 2007, mick_durkin at mail.berlios.de wrote:

> 3) a square wave of 1 second repetition rate (1 Hz).
> The two transitions are filtered against the timestamp
> of their arrival time so that the one nearest to
> the full second is chosen and it marks the second.

I see a big problem with this edge selection mechanism.  With many of
the gps we have seen it is not possible to get within 500mS of correct
time using only the time data from the serial stream.  So this will
result in randomly picking an edge, not picking the best edge.

A good example is my garmin GPS-18/LVC.  Grabbing time with garmin
binary results in a delay of about 430mS from the PPS.  So this code
will most likely lock in on the wrong edge.  When I try getting time off
NMEA at 9600 baud I see jitter of 400mS or more.  So then the edge to
lock on would be random.

Since my Garmin uses a pulse I can not test this third scenario but
you should try setting your GPS to different baud rates and then
comparing to other time sources to see how robust this system is.
