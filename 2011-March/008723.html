Greg Troxel <gdt at ir.bbn.com>:
> Without thinking too much, my reaction is that any real use will end up
> needing a select loop because other things are happening, and that
> telling people they have to use select and not trying to accomodate
> people who want blocking interfaces is probably the simplest path.

You argue a plausible theoretical case here, but it happens that with the new
gps_waiting() *none* of the test clients require a select client any more.
Here's how it breaks down:

gpxlogger, lcdgps: Don't take input.

cgps: Takes input.  Handled by doing a nonblocking wgetch() after each
gps_read().  

xgps: Takes input.  The effect of select is implicit through the GTK
event-handling loop.

These could all be coded using the read-with-timeout.  And I think they
cover all the plausible interface patterns for a client.
