Excerpts from Eric Raymond's message of Wed Mar 09 21:40:39 +0100 2011:

> These could all be coded using the read-with-timeout.  And I think they
> cover all the plausible interface patterns for a client.

FWIW, this sounds fine with me, but please add a special value for
blocking forever. Some clients (e.g. a background logger that logs your
position whenever a GPS is attached) have no need for timing out. With
hardware like OLPC XOs [1] that can suspend the main CPU and automatically
wake up on any external event, even "long-term" timeouts can reduce
battery life significantly (imagine ten different tasks each waking the
machine once an hour while it's sitting idle on the desk).

You could also use uint64_t or reduce the precision to 1 milliseconds
(waking up once every 1.5 months), but using a single, special value
to indicate an infinite timeout would be a better fit IMO.
