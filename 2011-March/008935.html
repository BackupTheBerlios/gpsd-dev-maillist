Hm.  The client tests probably aren't in a form you can use.  Thanks for
bringing this up, it raises some questions about why that is.

When you speak of a client regression test, it seems to me you could
mean two different things: (1) a regression test for a language-binding library,
or (2) an end-to-end regression rest of an actual client.

In general, a test case for a binding to language Z requires some code that 
generates a textual report from the native language Z data structures (after the
JSON unpacker in the binding has done its thing).  That's sufficient; you can
use it to generate checkfiles, then use regress-driver ora similar tool for
regressions.

There's such a test for the C-binding library; it's in the libgps
TESTMAIN code.  I don't have an analog for the Python-binding library,
and probably should.

A *client* regression is in general a much more difficult thing, as
you'd have to capture the output events from the client.  Easy only
if the client is a non-interactive textual dumper a la gpxlogger;
otherwise (in particular if the client is GUI- or cuses-based) there
are significant technical problems.

I think the right answer is that I need to make a client-binding test part
of the delivery requirements for new bindings.  After I clean up my own
act by writing a Python one.

(Presently I test the Python binding by firing up xgps on live data and
eyeballing it to see id it looks broken.)
