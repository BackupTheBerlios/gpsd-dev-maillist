On Thu, Mar 24, 2011 at 10:55 PM, Eric Raymond <esr at snark.thyrsus.com> wrote:

> Part of the reason I think this is an interesting technique is that I
> believe the following: on essentially all modern processors, memcpy(3)
> is optimized to a single-instruction bitblt that reads and updates
> memory in a monotonic order (certainly this is true of naive C
> implementations of memcpy()). Thus, it should be sufficient for the
> writer to memcpy() into the struct and the reader to memcpy() out of
> it.  It doesn't matter whether or not memcpy() is interruptible.

The instruction(s) would be REP MOVS on the x86 architectures:

http://siyobik.info/index.php?module=x86&id=279

Other architectures, particularly RISC based do not have such instructions.

>From the  REP instruction page:

A repeating string operation can be suspended by an exception or
interrupt. When this happens, the state of the registers is preserved
to allow the string operation to be resumed upon a return from the
exception or interrupt handler.

BTW - if anyone can tell me how to optimize the library calls in
<string.h> so they disappear from the binary, I'd be grateful (and
could answer a now-stale question on it on stack overflow).
