>
> If I may suggest something here:
>
> Why not have
>
>    ?WATCH={"enable":true,"json":true,"classes":["TPV",...]}
>
> and
>
>    ?POLL={"classes":["TPV",...]}
>
> ? The list of requested classes should be a JSON list, not an object. If
> the "classes": part is missing, default to all classes. This way, the
> filtering logic could be the same for both requests.
>
> Just an idea...

Exactly.  WATCH and POLL should converge as much as possible.  They
are very close now since the TPV and SKY objects appear.  No one has
yet discussed the GST or ATT.  (The DoP data seems like it should be
in the GST instead of the SKY object and the ATT looks like it could
use a bit of cleanup - maybe create a subarray or subclass for the
triplets - x,y,z or pitch/roll/yaw - are there compasses that only do
x,y?).

The rest is how to fit the POLL response into a fixed buffer size and
how to format it the best way so that the result is one object but
easy to parse.

Also, if I'm using a SBAS/WAAS enabled GPS so it is already correcting
for the ionosphere (and the status reflects it), will a RTCM stream
also attempt to correct it or not?  Will it result in a
double-correction?
