I will concede that point to the extent that obscurity isn't a
security policy, however I don't like to make it easy to try to break
things.

Users and clients have no business knowing file paths either.  Except
for the tcp/udp name collision problem (only if you have the IPs and
ports on both ends are they unique), I wouldn't bother about a hash of
the path even as a string.  At least that would be constrained and
wouldn't convey information which could be used or misused by clients.
 (well, you could search /dev and hash each path...)

Strings are messier than other data types.  Right now the other
discussion is the length of the buffer when it is first set - and this
messiness will be broadcast in all the JSON messages affecting the
size of those too.  An int or uint cuts those problems out at the
start (either index or file descriptor).
