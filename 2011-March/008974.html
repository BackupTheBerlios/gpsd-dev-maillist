Yo Eric!

On Tue, 22 Mar 2011, Eric Raymond wrote:

> > The only way around this is a memory barrier, which is an instruction to
> > the CPU to flush its cache before continuing.
>
> Ah, I see. Your "memory barrier" is a way around the problem of the
> hardware-level memory controller on multicore system not enforcing cache
> coherency itself.

Yep.

> I agree that this is a real problem in general.

My son likes a lot of old DOS games, and many will only run in a VM
set to single core.

>  I'm not very worried
> about it in the use case we have for a shared-memory interface,
> though.

And it has not been a serious problem for ntpd yet.

>   I anticipate this interface being used in low-power,
> single-core deployments like tz's Chumby.  I'd be OK with adding dire
> "Don't Go There On Multicore!" warnings to the docs, if it came to that.

So when Chumby goes Dual Core things break.  Even cell phones are dual
core now.

> Anyway, since we need to solve this problem for ntpshm, whatever
> way of setting up memory barriers we develop for that case can be
> re-used here.

OTOH, instead of reinventing the wheel we could try to find a
good library (not pthreads).  pthreads requires all threads be in the
same process.  For example I recently mentioned this one:

	http://uw714doc.sco.com/en/man/html.3synch/Intro.3synch.html

Sadly that one is not standard.

The ntpd folks are hot on the trail, so I'll watch what they come up with.
