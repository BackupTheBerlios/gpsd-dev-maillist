On Tuesday 28 March 2006 16:39, Chris Kuethe wrote:
> I agree we can't/shouldn't make XML the default protocol: that could
> break existing, non-xml-aware apps. I really like gpsd's one-line
> plaintext protocol, and i don't see a huge value in deviating from
> that style. After all, the whole point of the gpsd protocol is to
> insulate you from hardware-dependent aspects of GPS and to give you a
> more sane protocol to move data around in. I am open to the concept of
> optional XML output. Then again, we have most of it already:
> gpxlogger.
>
> I'm less concerned about storage formats... i store gps data in csv,
> text, html, xml, sql... heck, i could probably hack up something based
> around AVI containers. there are, as you point out, tons of libraries
> to parse these things.
>
> For gpsd protocol v3, i'm open to completely breaking backward
> compatibility. send the "+" or "*" command which tries to switch to
> the extended protocol, which then can have all kinds of other goodies.
>

IIRC, there is a way to negotiate up in the current protocol. Some kind of a 
version request that opens up a completely new name space.

I know that there is no policy encoded that forces any particular protocol. It 
could be xml. OTOH, the current gpsd protocol has served us for around a 
decade. The current problem with it is only the narrow name space. Why not
just extend what worked, and hope for another 10 years. With current processor
power a format that uses the same single line format with just words instead 
of single letters is very easy to handle.

Providing backwards compatibility is cheap, and will keep old clients happy.
I believe xml is serious overkill. I could have used ASN.1 when I first wrote 
gpsd. It was hip then. You all would have hated me.

Please let me know, if you can't find out about the protocol promotion.
I know, I wrote it. Just don't remember the details. Eric might have some 
recollection.
