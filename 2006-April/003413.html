>  > However, those cases only occur in the event of either (a) GPS unit
>>  partial failure [connection good, output corrupted] or (b) manual

>>  circumstances.
>
>I (amicably) disagree with the admonishment associated with point '(b)'. With
>my Garmin 'GPS 35 TracPak' (UI-less, i.e. "hockey-puck") receiver, you
>actually *have to* "screw around with NMEA sentence output in the middle of
>active GPS use". That's the only way you can change the GPS' NMEA output, by
>sending an NMEA message to the GPS while it's powered up (and therefore
>transmitting useful NMEA data).

Point taken.  I should clarify my comment a little (and I accept that 
the admonishment was misplaced).  What I really meant was that if you 
have a client actually *depending* on the output of the GPS device at 
the same time as you change the output that device is sending, then 
you're generally asking for trouble.  Such a user should certainly 
know better than to remove an essential sentence on which that client 
(in this case, gpsd) depends.

In light of the above, is it realistic to suggest that gpsd should 
handle the case where the operator turns off an essential NMEA 
sentence mid-stream?  If so, I think we have to return to playing 
with the individual timeouts idea, which sounds very complicated and 
scary.  However, I can't see any compelling reason why we *should* 
handle that case: if you know enough to turn off essential sentences 
mid-use, then you should know enough to understand that gpsd can't 
give a reliable (insert parameter only found in that one sentence 
here) after you do so.

We await ESR's essay with interest!
