Gleb Smirnoff <glebius at glebius.int.ru> writes:

> I think this tasklet should be done within the gpsd project framework,
> without forking a separate project for say gpxlogger-2.

Absolutely.  I don't think anyone is opposed to having gpsd grow a
separate gpxloggerd.  My point was that gpxlogger fills a need now and
has very minimal dependencies, and we should make changes in such a way
that the original behavior remains available, with spiffier features on
platforms where that works.

  I am doing openlog() with LOG_PERROR, so all messages are printed to
  stderr, if we are running non-daemon.

If I run gpxlogger as a user, I don't necessarily want to have that end
up in the system log.

I wonder if having the parse/processing routines pulled out to a
gpxlogger_common.c file and having a non-daemon driver and a daemon
driver would make sense.  But leaving printf/syslog with the syslog
#ifdef'd on having openlog would work too.

questions we should be clear on (I'm not quite):

1. Does gpsd and gpxlogger work on cygwin now?
2. Do we care?
3. Does cygwin have openlog/syslog?

4. Any other platforms where the above 3 Qs are relevant?
