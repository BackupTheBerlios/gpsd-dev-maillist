Michael Ledford <michael at ledford.cc> writes:

> On Jul 5, 2010, at 2:46 PM, Gary E. Miller wrote:
>
>> pkg-config for OS X is trivially available from MacPorts.  hard to
>> believe
>> anyone compiling on OS X does not have MacPorts installed.
>
> Call me old fashioned. When something doesn't absolutely require
> something to build then I find no reason to install it. If the project
> managers wish to flat out require pkg-config, which in a way they have
> it is documented, that's fine. I'm simply appealing to the notion that
> if it's only required because you want to build something in QT should
> it still be required if you wish to not build the QT lib?
>
> Therefore I find no reason why if someone on Mac OS X wants to
> download the source tree and build it should go through the hassle of
> installing pkg-config from MacPorts or from scratch just to build gpsd
> when for some cases it isn't absolutely required.

On one hand that argument sounds reasonable.

On the other hand what pkg-config is is a standard way to express
dependencies.  It's not about qt, or gtk, or any other particular
program, but a replacement for each package having a hand-written
foo-config and autoconf macro.  So requiring that pkg-config exist for
people who are building from git (not *from source*, but *from git*, as
opposed to from a distribution traball) seems just as reasonable in 2010
as requiring autoconf in 2000.

It's arguably a bug in xcode that pkg-config is not present.  pkg-config
has been highly normal for several years.

It's a fair complaint that this should be documented.

(I should add that I'm writing from the perspective of a NetBSD/pkgsrc
user, and I often object to gratuitous requirements that things be like
they are on linux.  but pkg-config isn't a linux-only thing.)
