On Wed, Jun 1, 2011 at 8:38 AM, Eckhart Wörner <ewoerner at kde.org> wrote:
> (shameless advertising:) I agree with you that starting gpsd on demand is a
> good thing. If you want that feature, an init system which supports socket
> activation is the way to go. Coincidentally gpsd already gained support for
> systemd socket activation. ;-)

This is only partially helpful.  Most GPS units need to be on for some
number of seconds to get a lock, so "on demand" will generally not get
good position data for 30 seconds or longer depending on the state of
the GPS itself.  Many stay asleep until the USB port is opened.

As far as I can tell, GPSD only starts to see if there is something
which can provide GPS data when something connects to it and asks for
data.  With the right units this could be appropriate, but for most it
should at least verify that there is hardware and at least verify the
connection (and maybe a fix, and maybe keep it from sleeping).

Even NMEA is a nightmarish blob - power on or attach (bluetooth) and
you might get a stream - and it might be at any baud rate, and there
may be proprietary sentences and various quirks.  Then there are all
the binary protocols.

Some of the USB devices might be nice enough to have vendor/product
IDs that distinguish what it is.  But Bluetooth devices need to be
paired and the UI handling for the stack is broken or nonexistent
(such that turning on the GPS device would automatically connect and
start feeding the GPS stream).  I can do it by editing config files,
but nothing automatic.
