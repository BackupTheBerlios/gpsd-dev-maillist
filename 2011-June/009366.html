Ed W <lists at wildgooses.com>:
> Note I didn't say that you could do without a scripting language for
> some other tasks, only that the udev stuff currently appears to be
> fairly straightforward to implement in C?

It might be.  What I'm not hearing, yet, is a good reason to accept the
additional maintainance costs and increase in expected defect rates
implied by moving that code from Python to C.
 
> I think the UDEV part is arguably "necessary" for a complete system, but
> I can't think of any other currently python pieces which are necessary
> to have a running gpsd service?  ie if you changed the udev piece then
> python would be completely optional for many use cases (such as mine)
> 
> Hence it would be desirable to have a non scripting piece here (or at
> least limited to only posix shell)?

That depends on whether there is a significant class of deployments
that need hotplug but can't support Python.  I'm skeptical about this;
normally, embedded systems have hardwired ports and a set of sensors
that is fixed by the system integrator.

Do you have an actual use case like this, or is your objection theoretical?

> That said, if the most important goals were installation size,
> portability and speed, then I think lua is a clear winner (low few 100KB
> installation size and fast portable vm).

Agreed, despite some odd quirks like 1-origin indexing.  If lua had been
mature enough by around 2005 I might very well have a "prefer lua" policy
for this project rather than a "prefer Python" one.
