>>> What would you replace it with?  The package needs *some* scripting language.
>>
>> If we talk specifically about the udev stuff, then "why?"
>>
>> The udev script forks and then talks briefly to a socket - something

> 
> I don't know what sort of getting in the way Diego is referring to.  
> Perhaps he'll clarify.
> 
> The package needs a scripting language (really, a language with
> garbage collection and variable-extent types) because it contains a
> number of programs for which C would be a poor choice, imposing 
> a lot of low-level resource management complexity on code that
> doesn't have performance requirements hard enough to justify it.

Note I didn't say that you could do without a scripting language for
some other tasks, only that the udev stuff currently appears to be
fairly straightforward to implement in C?

I think the UDEV part is arguably "necessary" for a complete system, but
I can't think of any other currently python pieces which are necessary
to have a running gpsd service?  ie if you changed the udev piece then
python would be completely optional for many use cases (such as mine)

Hence it would be desirable to have a non scripting piece here (or at
least limited to only posix shell)?


> We use Python because I like Python. I'd like Lisp better, maybe, if
> there were one with good enough cross-platform support.  I'm open to
> rational argument about replacing Python, but the replacement would
> have to have some pretty compelling advantage to pay for the 
> conversion overhead.

Every scripting language has pros and cons. Familiarity is at least one
of the best reasons to pick one over the other.

That said, if the most important goals were installation size,
portability and speed, then I think lua is a clear winner (low few 100KB
installation size and fast portable vm).  However, I don't see any
reason to change from Python for anything other than the udev part
(which I would argue is near essential)
