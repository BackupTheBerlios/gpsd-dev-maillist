Michael Cook <mcook at bbn.com>:
> Should these LOG_... symbols become a formal part of the libgpsd
> API? Currently, the documentation (libgpsd.xml) says only this:
> 
>    <function>gpsd_report()</function>.
>    The library will use this to issue ordinary status messages.  Use
>    first argument of 0 for errors, 1 for ordinary status messages,
>    and 2 or higher for debugging messages.</para>
> 
> which means users of libgpsd can't write a gpsd_report like what now
> exists in gpsctl.c and gpsd.c, not without relying on undocumented
> symbols. (I ran into this very problem a couple weeks ago when
> trying to translate gpsd_report's errorlevel values into my system's
> logging facility...)

Well, you're correct that this documentation is out of date and needs
to be fixed, thanks.  On the other hand, I'm not sure any purpose 
would be served by documenting the levels and thus freezing them in
place - as opposed to saying "Go look at gpsd.h".  If you're deep enough
into the code to be writing a gpsd_report() implementation, you're
going to have to do that anyway.

Nore that clients no longer need to define this function; now, you need ro
be messing with libgpsd before it becomes an issue.
