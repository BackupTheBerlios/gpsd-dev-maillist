So that's pretty disconnected from near-term reality.  As an example, I
think NetBSD-current has gone to 64-bit time_t, so as long as migration
happens well before 2038 we should be ok.

> So the question is do we want to support those large dates? If not  
> then the tests should be modified. Otherwise the patch I provided  
> should be applied to support them.

It's not just the tests; to support dates > 31 bits we'd have to define
a local time_t and use it instead, and this is a huge undertaking.  I
don't see what it buys us - casting in the test makes the test pass, but
wouldn't make straightforward code using time_t work, so the test
wouldn't reflrect reality.
