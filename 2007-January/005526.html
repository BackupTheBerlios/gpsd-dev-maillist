> To be useful, I'm afraid each driver might have to decode whatever the
> receiver gives it into a prettier struct.

   Yep, sorry I did not state that explicitly.  That is what my rinex.h
patch attempts to provide, a common data structure where you can store
normalised observation and ephemeris data for latter transmission to the
user (which does not have to be in RINEX format).  I merely copied the
RINEX schema because I thought it best not to try and reinvent the wheel
as to what is needed and how to represent it.

   Having said that, I do want gpsd to speak RINEX :)

   With regards to the discussion as to whether it should come from gpsd
itself or some external client, let me see if I get this right.  What
you propose would look something like:

gpsd --- (GPSD format) ---> client [ PVT data ]
gpsd --- (NMEA format) ---> client [ PVT data ]
gpsd --- (RTCM format) ---> client [ DGPS data ]
gpsd --- (format 'X') ---> gpsd2rinex --- (RINEX) ---> client [ RINEX data ]

   as opposed to:

gpsd --- (GPSD format) ---> client [ PVT data ]
gpsd --- (NMEA format) ---> client [ PVT data ]
gpsd --- (RTCM format) ---> client [ DGPS data ]
gpsd --- (RINEX format) ---> client [ pseudoranges / nav. data ]

   or perhaps I have made totally the wrong assumptions as to the idea
you mentioned?  Was it perhaps some sort of "plug-in" approach you were
describing?
