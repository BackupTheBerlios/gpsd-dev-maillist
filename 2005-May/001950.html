Yo Eric!

splint also complains if I pass an (unsigned char) to a function
expecting an (unsigned int).  I can make a case for bitching about the
reverse, but not the forward condition.  Is there some way to tell
splint that it is OK to put a smaller type in a bigger type?

Here is another silly one:

	double track = 0;
	if ( track < 0 ) {

garmin.c:300:10: Dangerous comparison involving double types: track < 0
  Two real (float, double, or long double) values are compared directly using a
  C primitive. This may produce unexpected results since floating point
  representations are inexact. Instead, compare the difference to FLT_EPSILON
  or DBL_EPSILON. (Use -realcompare to inhibit warning)

Even gcc is smart enough to implement this test by looking at the sign bit
in the double.

There is a way to silence this silliness in .splintrc: -realcompare.  Any
idea how to fix this inline?

The man page says nothing about -realcompare or much of anything else
usefull.
