On Mon, 30 May 2005, Rob Janssen wrote:

> My guess is that on a typical P4-class CPU, on wich execution speed
> depends more on the execution path than on the actually executed
> instructions, the multiply will be orders of magnitude faster.

So you think the glibc people put in all thos optimizations to make it
slower? :-)

I agree with you when talking about integer and branch code.  FPU code is
a very different beast.  A P4 class CPU may executre more than
one non-FPU instruction per clock, It may still take hundreds of clocks
to perform heavy FPU ops like fdiv.

> When we would want to help embedded systems without floating point
> processor, we probably could make a lot of difference by removing all
> floating point and replacing it with fixed point (scaled) integer
> arithmetic.  Or by going back to float instead of double.

Let them help themselves.  These sorts of optimizations are very system
specific.  It is very common for embedded folks to invent their own
float format to give them just the balance of speed/accuracty they need.
The only thing we could do -- and they have not asked -- is to use
arbitrary types for the variables.  Then someone without an FPU could
change the #define from double to whatever works best for him.

> ... warning that the order of evaluation is undefined and so it isn't
> defined in which order the pow() calls are made.  Of course this is true,
> but it is almost completely pointless.

No argument here.
