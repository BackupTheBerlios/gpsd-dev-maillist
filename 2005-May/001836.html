Eric S. Raymond wrote:
> 
> I'm merging this with one API change.  It's just a bit too tricky to
> have the set_callback function replace a *synchronous* callback if one
> has already been set -- instead, it now returns an error indication in
> that case (and the documentation warns this is an error).  

In man page it is still said that calling gps_set_callback() twice will 
replace the current callback function, however with the API change this 
is no longer true (callback uses the gps_data->raw_hook, so the second 
time it is called it finds raw_hook already "busy", as in synchronous mode).
Moreover, if the raw_hook is not null, could you consider returning not 
0, say -1? In fact pthread_create() returns 0 in case of success, and 
returning 0 also for an error may confuse the user (although this 
behaviour is clearly explained in the man page).

I'm also shipping a new correted man page for libgpsmm (method names 
were slightly worng). I'm sorry I didn't catch this man page errors 
before the 2.25 release...
