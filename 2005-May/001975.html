It seems to me that this problem is simply too much machine and compiler
dependant. So we may only hope in gcc's optimizing capabilities (if gcc
is used)...

> alternatively, we could write:
> 
> z = sqrt(x*x + y*y);

I still believe all these workarounds only give us worse code readability.
If we really think that a totally "splint clean" capability is
important, we may consider spending some time configuring it exactly how
we need it to work.
At the beginning we started changing the "flavor" (style) of the code to
make splint happy, now we make splint understand how we like the code...
and we should continue "teach" splint how to read our code...

gpsd is powerful, and it is more powerful because it is BSD licensed and
its code is quite simple to understand. I don't think many
/*@comments@*/ like this wold make it more powerful, and we should
consider the negative weight of these comments against the discover of a
few bugs which are also difficult to happen.
It may be boring, but maybe hand checking twice the code could save us
from the splint's unforgiving eye! ;-)
