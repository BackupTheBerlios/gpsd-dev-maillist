If the preferred method is to NOT start gpsd at startup, then how  
does Linux handle Bluetooth GPS's or pure serial GPS's?  I've used  
hotplug under Linux, but haven't looked at the details.  How does  
hotplug see serial & Bluetooth serial devices become attached?  All  
my Bluethooth serial experiences (Linux, Windows, PocketPC, Palm OS,  
Mac OS X), indicates that the Bluetooth Serial connection is NOT  
created until an application opens the "virtual" Bluetooth Serial  
device......

launchd does support starting a daemon when an application needs the  
services (like inetd or xinetd).  I was worried about the application 
(s) not being patient enough to wait for the daemon to start and  
initialize.....  Then the application would have to be told to "look  
for the GPS" again.  I haven't tested this yet.  Would it be  
preferable to start it in this method?  Are the apps you're aware of,  
patient enough to wait for the daemon to start?

If we can decide on a "best" method to start gpsd for Mac OS 10, I'd  
be willing to write and maintain the Mac specific docs & scripts.  I  
could provide support for Mac OS X 10.3 & 10.4.  At least on the Mac,  
I'm still leaning toward gpsd startup on boot-up.

> Currently, make install does not do the equivalent of this (which is
> installing the hotplug scripts) under Linux.  That is left to the RPM.
> The intention is that make install does the distro-independent part
> (putting the gpsd and gpsd-client binaries in the command path) and
> that there be a distro-specific installation step fopr each distro.
>
> launchd is a prett good idea, but it is not necessarily the right  
> way to
> launch gpsd.  It would be better to not start gpsd at boot time, but
> instead to start it when the first USB device that you identify as a
> GPS goes active.  This is what the Linux hotplug scripts do.  They
> also take care of telling gpsd when to add a GPS device to its
> internal list of devices, and when to do removes.
