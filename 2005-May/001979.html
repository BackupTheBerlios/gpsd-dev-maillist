On Tue, May 31, 2005 at 01:19:54AM +0200, Rob Janssen wrote:

> I have advocated (and done) hand checking a couple of times.  I
> completely agree with you that it is a better idea than trying to
> massage the code until an automatic checker outputs no warnings
> anymore.

> We have seen some cases of incorrect scaling and reading tonights
> mail there seems to be a new one.  Those errors are unlikely to be
> caught by a tool.

> What I worry about is that carefully coded and checked source is
> being global-replaced by constructs that may satisfy splint but are
> not working correctly.

  I'm not a huge fan of tremendous amounts of "baggage" on a project
that is, at it's core, a "fun" project (as opposed to something lives
depend on).  At the same time, I wonder if the addition of a very
small amount of structure in the form of automated tests might be in
order.  This could be very very simple, for example:

  1.  A "known bugs" file, where each bug gets a number (ie, #1, #2,
      #3).  A sentence or two describes the bug.

  2.  For each bug, a comment in the code noting that "Fixed Bug #42
      here 'cause I was accidentally casting a char to a pointer to an
      integer array".

  3.  Create a series of tests (perhaps run by 'make test') that tests
      each new bug as it's fixed (ie, link the the appropriate library,
      make the API call, and check the return value).  Maybe even some
      raw NMEA/Sirf/Garmin data to feed to 'gpsfake', then test the
      output.

  I don't want to suggest we take the fun out of gpsd, but this seems
pretty low-overhead.
