I had the pleasure to have to support windows on a project a few years ago. It 
was a server not unlike gpsd. Windows has no select()/poll(), but an api that 
lets you do similar things. Just diffrent enough so that the whole code 
structure would not easily be able to support it.

This is where Cygwin came in. What a s**t fest that became. Undocumented 
limitations in its implementation of select(), subtle bugs, bugs that had 
been known since ages, but never fixed...

A friend of mine is working with Cygwin now. I can not quote him here, bad 
language. 

I wish luck to anybody who tries, but something like this would almost have to 
be a fork.

On Friday 06 May 2005 02:37, Rob Janssen wrote:
> Chatting with some Windows users, I found that there could be some demand
> for gpsd in those circles as well...
> I presumed that there would be a "Microsoft GPS API" that applications use
> to talk to those USB receivers, but apparently there isn't.  Applications
> talk to a COM port (maybe a virtual port that appears when a USB device is
> plugged in) and directly read NMEA data.
>
> So, when you want to use more than one application at the same time you
> are out of luck.
> I figured it should be possible to make gpsd run under Windows, when
> someone picks that up.
>
> Considering the existing apps, ideally the daemon would create some
> "virtual COM ports" that will output NMEA data when connected.  However
> that will probably require some in-depth Windows knowledge.
> Alternatively, one could use a separate application that provides the
> "virtual COM port to TCP connection" mapping (these exist for accessing
> dialout modems located on access servers, for example) and let that
> connect to gpsd via TCP.
> It would help when such a session started with R=1 by default.
>
> Any thoughts?  Or are we simply against any Windows support?
