Gary E. Miller <gem at rellim.com>:
> The delays needed in baud changing ar also problematic.  I really hate
> to do things like add large wait delays in a dameon that has to serve
> multiple clients.

Don't worry about it.  The B command is disabled when there's more than one
user in order to prevent DOS attacks.  So large wait delays aren't really
a big deal here -- you can only be hit by one if you asked for it :-).

> I guess we'll see if we can solve these problems.  Please do not delete
> garminctl until we do!

I hadn't planned to, but I'm not going to ship garminctl/sirfctl/itraxctl
in the RPM, either, nor expose their docs on the website.  If you're defending
these on the theory that they're scaffolding, they won't be treated as part
of the public toolset.
 
> These standalone tools will always be useful bootstraps.  The real
> question is how long they need to persist.

And, as the project's software-architecture Nazi, I'm going to be the
guy constantly pushing for them to get scrapped.

> And having one big packet engine has maintenance problems of it's own.
> Standalone tools don't affect as many different GPS so do not need to
> undergo as extensive testing as things in gpd.

I'm aware of this problem. That's why the packet getter has its own
regression test.  You can help by adding more sample-packet initializers
(both well-formed and broken in creative ways) to packet_test.c

> As Chris pointed out, gpsd can have a hard time recovering from all
> psossible configurations.  Using a tool that starts with the knowledge of
> the target GPS has a better chance getting the GPS into a better state.

You both say this, but I don't see what makes it true.
