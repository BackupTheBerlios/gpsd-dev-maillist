> > Not quite. There are a number of quantities that are split: 11 bits

> > subframe.c
>
> Well, when you get this working, nuke subframe.c with my blessing.

Why? Subframe.c has one job to do: find the leap second offset. That's
a valuable number. We use it. Code that isn't used to do the main job
of gpsd (or a substantial number of accompanying  utilities) doesn't
need to be in libgps, but code that directly supports gpsd can stay.

Perhaps a re-reading of IS-GPS-200D is in order... to see when they
transmit the upcoming leap-second warning. If it's always in the 12
hours before Jan1 or Jul1, we could turn on subframe decoding, check
if they're going to add a leap second, set a couple of flags to say
whether we checked and whether there is going to be a leap second, and
then shut down the code path for another 6 months.

This is valuable to those of us with long-running gpsd instances. I've
got a 310MB log of two weeks of SiRF messages for example.
