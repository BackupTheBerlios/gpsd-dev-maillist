On Mon, 30 Oct 2006, Eric S. Raymond wrote:

> Gary E. Miller <gem at rellim.com>:
> > > +*** Merge cgps and xgps
> > > +
> > > +cgps and xgps should merge into a single test client that honors the
> > > +GPSD_UNITS environment variable and chooses its UI mode depending on
> > > +whether or not it's running under X.  A good small project for someone
> > > +who doesn't want to get into the full complexity of the daemon code.
> >
> > This would then require small systems to either have the X libs installed
> > or be compiled without X.  Certainly there is code that can be shared
> > but there is real value in not cluttering commandline tools with all
> > the Xlibs.
>
> If this were 1994 and Linux had only static linking, I'd agree with you.
> But I don't think this is a real issue any more, not on modern systems
> with dynamic linking.  We have two major use cases:

You miss my common case.  Where X is run on a remote server but due to
latency you do not want to run a tool using X.  At a minimum you need a
command line to quash the X mode and when you forget you flood your slow
circuit with X packets until you can get a ^C back to the remote host.

"mtr" (Matt's Traceroute) does what you suggest and it drives sysadmins nuts.
I have lost much time trying to explain its dual nature to folks that
are not X wizards.  Users are always surprised and baffled as to why it
will chose curses over X, or not.

You can not just move a combined binary from an X system to a non-X
system.  I do this with cgps all the time to check versions.

Even if the libc is the same version you can not copy a binary from an
X7.0 to an X7.1 system or vice versa.

I tend to think that UNIX prefers small programs that do one thing well
over big programs that have everthing but the kitchen sink in them.
