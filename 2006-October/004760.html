(Copied to gpsd-dev as I think this point is of general interest.)

Gary E. Miller <gem at rellim.com>:
> > +*** Merge cgps and xgps
> > +
> > +cgps and xgps should merge into a single test client that honors the
> > +GPSD_UNITS environment variable and chooses its UI mode depending on
> > +whether or not it's running under X.  A good small project for someone
> > +who doesn't want to get into the full complexity of the daemon code.
> 
> This would then require small systems to either have the X libs installed
> or be compiled without X.  Certainly there is code that can be shared
> but there is real value in not cluttering commandline tools with all
> the Xlibs.

If this were 1994 and Linux had only static linking, I'd agree with you.
But I don't think this is a real issue any more, not on modern systems
with dynamic linking.  We have two major use cases:

1. Tiny systems that can't support X at all.

2. PCs where X applications are running essentially all the time.

For case 1, we honor --without-x as Chris pointed out we should,  Nobody
pays any X library overhead at all under any circumstances.

For case 2, the X libs are going to be in core all the time anyway, so
the only overhead a combination client would have over "plain" cgps is
memory space for the hatful of code presently in xgps.c and a couple
of dynamic-link pointers.

About the only time we could potentially have an issue is if somebody
happened to be running the merged client on a virtual console with
X shut down,  Even then, modern VM systems probably wouldn't pull the
X libs into the working set unless one of the entry pointers got chased.
