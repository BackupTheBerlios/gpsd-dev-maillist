Chris Kuethe <chris.kuethe at gmail.com>:
> Don't merge xgps and cgps. This will interfere with installing
> pre-packaged gpsd on systems without X.

Yeah, like *that's* ever going to happen.  For our purposes the only
'systems without X' these days are headless boxen or SBCs run by the
same people who are going to be using all our nifty configure options
to hand-tune a build from source.

Now I grant you this might not be the case if we were writing, say, a
point-of-sale system.  Those people really do feed bunches of
character-cell terminals with cgps-like curses clients; linking X libs
to one of those might be a bad idea, if the tuning on the underlying
VM sucks badly enough to pull the libs into the working set even
though the pointers to X never get chased.

But I think this is not a use case we need to worry about.

> We have something like gpsd-{core,clients,server,devel}? Why not have
> an xclients package. That means that on lighter systems that
> don't/won't have X you don't install anything that needs X, but you
> can still use all the curses goodies.

The gpsd-clients package is already X clients only.  cgps is in the
base RPM.

> If you must clean up rendundant clients, consider rolling cgps and
> cgpxlogger together... they both have to parse the same daemon
> message, so really the change is to add some sort of log output switch
> to cgps: gpsd messages, gpx, nmea...

That's a good idea too.
