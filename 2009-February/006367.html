On Fri, Feb 6, 2009 at 10:30 AM, Lukas Bischof <lukas.bischof at gmail.com> wrote:
> Hello folks,
>
> i am currently writing on my bachelor thesis. My topic is A-GPS. During
> my research i have found some specs and docs from the open mobile
> alliance - but no free implementation of the SUPL protocol. My bachelor
> thesis is nearly finished - but i want to implement a SUPL client, a
> SUPL server and a SUPL proxy on my own. So, is there any need for A-GPS
> in gpsd?

Is there a need? not really. Would it be a bad idea? probably not.

Loading initial assistance data into a receiver can generally be done
in-band, but we don't have the machinery in place to build the
receiver-specific assistance packets. Some receivers expect raw binary
ephemerides/almanac, others want it neatly broken out, some expect
different messages for orbital information and atmospheric modeling...
There is some support in gpsd for RTCM and NTRIP, though they probably
don't get much use... these protocols require receivers capable of
accepting input corrections, and that excludes most consumer
receivers.

I don't want to discourage you from hacking on A-GPS and gpsd
integration, I'm just not sure that we'll be all that helpful. But
this does sound interesting... keep us in the loop?
