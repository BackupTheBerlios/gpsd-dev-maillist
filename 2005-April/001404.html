Yo Rob!

On Fri, 15 Apr 2005, Rob Janssen wrote:

> > The problem is not just xgps, it is anything that uses the libgps.
>
> Do you mean you want to solve this in libgps?

Yes.  Mostly.

> Can't we just return metric data from libgps and leave the conversion into
> local units to the application?

Yes.  The request Eric made yesterday was that libgps have a function
that returns to the client a best guess at desired units.

> It is not like rocket science to make such conversions...  and
> applications are in the position to query and store user preferences.
> Libraries are not.

Agreed.  I see this as akin to the locale() function.  Query it if you
wish, then do what you wish with the data.  Simple programs like xpgs
can use it, smarter programs like gpsdrive can ignore it.

> More complicated conversions like computation of the geoid separation have
> a place in gpsd/libgps, but I think it should not be overdone by offering
> simple multiplicative conversions like metric->impereal.

Agreed, general rule of floating point is to touch the numbers as little
as possible lest those creeping inaccuracies get magnified.
