On Sat, 16 Apr 2005 10:26:13 +0200 (CEST)
"Rob Janssen" <gpsd at xs4all.nl> wrote:

> I cannot imagine that a large application that supports libgps, for
> example a wifi wardriving system, a radio amateur APRS program, or a road
> navigation program, would want to get such service from libgps.

I could not agree more. Having ligps use different units depending on the mood
of the day make writing applications mre difficult. This is because applications
need to process the received data and run some calculations. Thus the application
is likely to rely on one internal unit. libgps delivering units depending on the
environment would force the application to implement all possible conversions.

I apologize if that idea was already discussed, but I would suggest that, if unit
conversion is to be supplied at all, it be application driven, i.e. the application
code directs libgps what unit to use. That would be a plus compare to NMEA,
which tend to be a mess when it comes to units (some speeds in knots, elevation
in meter, etc..).

The alternative is for gpsd to just cleanup the NMEA mess and provide everything
in the standard unit system (i.e. metric).
