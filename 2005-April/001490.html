On Sat, 30 Apr 2005, Eric S. Raymond wrote:

> Gary E. Miller <gem at rellim.com>:
> > The shmget in ntpshm.c can open sthat segment fine either before or
> > after dropping root perms.  The problem is a seg fault when it
> > writes to the shared memory as nobody.
>
> That's bizarre.  Do you have any idea why?

Well, it seems natural to me that if the shm memory segment is marked
only read/write for root that when a program is not euid root that
they can not read it.  OTOH, it would seem natural that if you opened
it as root, then dropped euid root that you could still access it.

So I could make a case for either behavior.  Have not found any doc yet
to explain the behavior my tests observe.  I hope someone that understands
shm can shed some light.
