> Tilman Koschnick <til at subnetz.org>:
>> I have some connection problems with my Garmin eTrex (NMEA, serial
>> interface, attached to a PL2303 serial to USB converter).
>>
>> The problems seems to be with the packet sniffer when hunting for the
>> right port settings. Correct settings are speed 4800, 8N1. Most of the
>> time, gpsd latches on to that and everything is fine.
>>
>> Sometimes, it goes past it and ends up with 4800, 7N2. After receiving
>> some sentences, gpsd hangs (the last two lines only appear after I press
>> Ctrl-C):
>
> Can you capture a log from the device that will reliably cause gpsd to
> lock to 7N2 within gsfake?

Is there a record of receivers that really need this 7N2 mode and won't
work in 8N1?
Could we maybe devise another way of detecting such a receiver than trying
to receive and see if it locks?

There is no real difference between 8N1 and 7N2 except that the high bit
is always set when you receive data (and should be set when you send data
and the receiver really really wants to see it).
I would think that setting 8N1 and maybe looking for high-bit-always-set
is a better clue than just trying to set 7bit mode in the serial chip.
