I apologise for not being entirely clear in the paragraph quoted 
above, although I believe it made sense if you read the rest of the 
message carefully: my point is exactly that I also know of no 
complaints from such users either.  This is despite the fact that 
there are many out there, and what they are currently getting is NOT 
truly cleared-at-start-of-cycle data.  The logical conclusion is that 
either (a) there aren't any users out there who do that kind of 
logging with NMEA GPS units (unlikely), or (b) the users out there 
doing that kind of logging with NMEA GPS units don't care when they 
receive differing values for the same parameter with the same 
timestamp (due to the existing bug in the start-of-cycle flagging 
logic).

If we run with (b) - which I'd say is far more likely than (a) - then 
we can go ahead with my modification (default) of Eric's proposal (to 
add a command-line switch).

If (a) is the case, it is likely to remain that way, as the chances 
of a new user using an old GPS are slim indeed, especially if no such 
user currently has such a unit.  Even if they do get such a unit and 
have an application that requires CaSoS, they will have the option to 
turn it on.
