> Rob Janssen <gpsd at xs4all.nl>:
>> Maybe we could build optional XML message support (as a compile-time
>> option) first, and then work from there.  It could at least be used to
>> evaluate what the bloat really is, and if it really poses a problem.
>
> What would you want to do, write a push mode that ships GPX messages
> rather than O/Y?  I might be willing to go with that.

Well...  I did not yet investigate if a schema for GPS data already
exists, and GPX indeed looks like what I was thinking about.
(it expresses things in terms of waypoints and tracks while I was thinking
more about continuous pushing of fix information, but you could consider
those the same thing)

I still think using with XML leaves us (not only core developers but also
people who want to extend gpsd, accomodate their own special needs) with
more freedom.

We also could ship messages that contain precisely the information
received from the receiver in the message we are currently decoding
(indicated by the returned mask) without having to consider parser
limitations at the client side (which may force us to send dummy values
for items we cannot retrieve from the current message from some particular
receiver).

Adding something like the truenorth support or inclusion of other special
data that a receiver may make available (like temperature), without
breaking compatability with old clients, is much easier.
A suitably clever schema could allow addition of timestamps per value or
tuple, indicators for buffered vs recent data, etc.

Going with a "standard schema" of course again limits this freedom somewhat.

Rob
