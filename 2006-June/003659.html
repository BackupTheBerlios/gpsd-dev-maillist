Yo Robin!

On Thu, 8 Jun 2006, Robin L Darroch wrote:

> Default behaviour:
> A) Clear at Start of Cycle (pro: may work better for data loggers by avoiding
> mixing one-second-old data with zero-second-old data; con: introduces false
> jitter into output of certain parameters from NMEA data sources - likely to be
> preferred by certain more experienced users and those who want their data as
> "pure" as possible)

What is a cycle?  Not all gps provde data every second.  The NMEA
standard allows for much slower data cycles.  My now dead EC-10X uses
2 second cycles and even longer cycles are allowed by NMEA.  I suspect
some GPS units use very long cycles to save battery power.  Also not all
data is output every "cycle".  What is to be held for a "cycle"?  Just
PVT data or does this plan include sats and other data?

> B) Never Clear (pro: always provides good data when available from GPS; con:
> mixes one-second-old data with zero-second-old data when using NMEA data
> sources - likely to be preferred by novice users and application developers)

This one could get someone killed.  At some point, if there is no new
data, it had better be cleared.  As a pilot I would want this to be a
relatively short time, maybe a few seconds at most.  Even in a car
in an urban canyon I would rather have no data than the wrong data.

And most important, how does the client know if he is getting stale
or current data?  When a GPS is giving dead reckoning data there is
a flag to alert the user.  gpsd needs to differentiate or separate the
high quality data from the not so high quality data for apps that
care.
