> In order to avoid startup lag as much as possible, the sample size is
> set small -- small enough that just one corrupted header might lead to
> failure of lock.

Note that removing the "first try 8N1 then 7N2" from the detection
mechanism cuts the number of attempts you have to make in half...  this
should make room for longer intervals at each speed.

I should note that detecting Trimble TSIP is also a bit unreliable,
largely because the receiver is sending so little data until it is told
what to send.  I often pre-set the tty port to the correct baudrate before
starting gpsd.
Spending a little more time at each baudrate and moving the 8/7 detection
into the statemachine might help.
(TSIP is detected at 8N1 and then sets 8O1 because that is what it really is)
