Rob Janssen <gpsd at xs4all.nl>:
> > That's an interesting idea -- I mean, searching for N2 only when
> > we see an NMEA leader with high bit set -- but it wouldn't solve Til's
> > problem, which is a failure to lock on at 7N1 before it gets to 7N2.
> 
> When for some reason the first detection at 8N1 fails, it is not that
> strange that a second attempt at 7N2 will succeed.  Especially because the
> port is set in a mode where framing errors are not checked, so 8N1 input
> will be accepted in 7N2 mode and will just appear to have the top bit
> masked away.

I understand that. But if Til is still having this problem, the correct fix 
is probably to make the input sample size larger so the state machine has
more chances to sync up.  My best guess about what's going on is that his 
line is a bit noisy and occasionally corrupts bits in an incoming packet 
leader.  

In order to avoid startup lag as much as possible, the sample size is
set small -- small enough that just one corrupted header might lead to
failure of lock.
