At 10:12 -0400 8/6/06, Eric S. Raymond wrote:
>Rob Janssen <gpsd at xs4all.nl>:
>>  I think gpsd should not do any interpolation, averaging or similar, but it
>>  should be able to buffer the last consistent fix from the receiver and
>>  make it available on request.
>
>I agree.  The thing we have to thrash out next is how to enable
>this behavior.

Hopefully not too much thrashing left there... and it HAS been 
productive, if only to separate the myth from the reality regarding 
the advantages and disadvantages of both approaches.

Only two questions remain as I see it:

Enabling:
1) compile time (./configure --clear-at-cycle-start or ./configure 
--never-clear)
or
2) command-line (as per my patch posted earlier today)

and

Default behaviour:
A) Clear at Start of Cycle (pro: may work better for data loggers by 
avoiding mixing one-second-old data with zero-second-old data; con: 
introduces false jitter into output of certain parameters from NMEA 
data sources - likely to be preferred by certain more experienced 
users and those who want their data as "pure" as possible)
B) Never Clear (pro: always provides good data when available from 
GPS; con: mixes one-second-old data with zero-second-old data when 
using NMEA data sources - likely to be preferred by novice users and 
application developers)

My vote is for 1+B.  My patch does 2+B.  I think 2+A is probably the 
worst option.  What do others think?
