>Robin L Darroch <robin at spade-men.com>:
>>  Attached is a patch to implement my amended proposal for implementing
>>  optional Clear-at-Start-of-Cycle logic.
>
>A compile-time option isn't good enough.  Different clients may have different
>smoothing requirements, especially when the server is mediating for more
>than one GPS.

But a compile-time option is still better than no option at all, 
which is what we have right now.  To put it simply, 
Clear-at-start-of-cycle isn't good enough.  It doesn't work.  Not for 
clients that want to use queries, for example.  I agree that in the 
long term, it will be better to EXPAND the compile-time setting 
capability with a per-client option, which is why I am allowing for 
"J" to become more than just a query command (it will become the 
per-client setting command).... but for now, we already know there 
are people out there who can't get the data they want out of GPSd, 
and there is no reason not to provide them with the OPTION to build 
their own GPSd the way they want it.

Compile-time is simple - and DONE... per-client is a significantly 
bigger ask, and while I'm willing to do it some day, I don't think we 
should wait on that because of a theoretical possibility that some 
server may need it some day: we have a REAL WORLD demand for CaSoC to 
be made optional right now... why the opposition to implementing the 
simple fix?
