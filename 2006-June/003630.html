> The altitude in invalidated because no altitude for the current timestamp
> has been seen yet.  To report an old altitude is a lie.  What is a client
> to do when the REAL altitude shows up on the current timestamp and it
> is different than the LAST altitude reported in the current timestamp.
> This will really confuse data logging programs or any program doing its
> own calculation on the data.
>
> If you want to smooth/average/hold data then the way to do it is in
> the client.  gpsd should not lie about the current data from the GPS.
> No data is no data and should be reported as such.

I don't agree with this strict standpoint.  One of the tasks of gpsd is to
simplify the access to a GPS receiver and the data it sends.  There is no
point in making it harder by requiring buffering in the client that is
purely required because of lowlevel protocol issues between gpsd and the
receiver, something the client should not have to be aware of.

I think gpsd should not do any interpolation, averaging or similar, but it
should be able to buffer the last consistent fix from the receiver and
make it available on request.
We can debate whether this should be standard or a switchable option (by
client command), and what facilities we provide to guarantee a consistent
result set and no stale data.  But sending NaN for some data because new
values for OTHER data have arrived is not a behaviour we should strive
for.
