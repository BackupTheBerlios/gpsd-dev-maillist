Yo Rob!

On Thu, 8 Jun 2006, Rob Janssen wrote:

> > And there is where we disagree.  A lot of folks use gpsd to gnerate trip
> > logs.
> > To have multiple records with the same time stamps and different data
> > in the fields is to create data that is confusing and contradictory.
>
> Who said that there will be records with the same timestamp and different
> data?

That was the initial problem definition.  At the top of the second the
GPS may send just the lat/lon.  To avoid waiting for an altitude that
may never come, gpsd ships off the new lat/lon, with the current timestamp
and NaN for the altitude.  A few tenths of a second later the altitude shows
up so gpsd sends that off with the lat/lon.

As I understnad it, the proposed solution to get rid of the fist NaN altitude
was to replace it with older altitude data.  Then when the current altitude
gets received it is sent off.  Then you end up with two time stamps and
the same altitude.

The other solution, long since rejecteded, was to collect all the data for
one time stamp, then output it when the next time stamp shows up in one nice
packet.  That adds a one, two or more second delay and is was quickly rejected.

We really have two problems here, one where the client is polling for data
and another when gpsd is streemaing data as it gets it.
