Greg Troxel <gdt at ir.bbn.com>:
> That's true (and an emacs bug to have used one command to mean "do what
> you think I mean" instead of commands for actions).

Hey.  I *designed* that interface!  Myself!  :-)

Seriously, in 1992 the idea that you could just do the next logical
thing in the version-control cycle was quite the conceptual
breakthrough.  I watched people have a "*Huh*?  That's brilliant!" 
reaction to it more than once.

Those were ancient days. In retrospect, my biggest mistake in
designing VC wasn't C-x v v, it was failing to foresee changesets.
When I originally wrote it, file-oriented SCCS and RCS was all there
was. This left the architecture of the mode in a bad state which
I only fixed in 2006.  Would have been sooner, but I spent a decade
being Mr. Famous Geek and a lot of projects got put on hold.

You are right that the logic of C-x v v leads to bad consequences 
in this instance. I think a better fix than wiring in knowledge
about file extensions might be to query the VCS to see if the file 
is ignored and refusing to register it if so.

>                                                  But I don't think
> that this is because of autoconf - it's because in general any
> build/config system has to find paths to where things actually are and
> then substitute them.

Point.  But I want to keep that stuff to a minimum rather than
deploying new substitutions at the drop of a bit. Bearing in mind
your cautions about the Python path problem, I'm going to be 
looking for ways to reduce those dependencies.
