Remco's first send of his message about protocol design mistakenly
went to me only.  This is my reply, which Remco suggested I post
to the list so we can continue the conversation on-list.

Remco Treffkorn <remco at rvt.com>:
> I put the L command in to open up a new name space.

Yes, I noticed that move.  It was my first clue that, despite the
amount of dust on the code when I found it, you actually have good
design instincts.   Which, in turn, is a major reason I stuck around to
improve the code rather than writing it off as a bad job.

> What I really wanted to do, was to put a tiny basic interpreter or another 
> script language in, but I have not thought that through yet. There are a 
> number of pitfalls.

I like this kind of hack.  But I haven't seen a good reason to embed a
language in gpsd yet.  I'm taking it in the opposite direction,
actually -- simplifying it so that applications will almost always
want timestamped 'o' tuples via watcher mode, and only supporting the
query commands as an evolutionary vestige.

There is something related I'd love to do, though -- which is wrap the
libgpsd library so it becomes a Python extension module.  If I could get 
that far, I'd think very seriously about scrapping gpsd.c and reimplementing
the daemon main loop in Python.

> Are there any ideas about how the protocol should look like? Obviously, my 
> example is not a suggestion, I only wanted to explain the mechanism I had in 
> mind to go to another name space.

What do you want the extended protocol to *do*?
