On Thu, 24 Mar 2005, Chris Kuethe wrote:

> > I guess I did not make all the steps clear.  The leap seconds are in
> > the current ephemeris.
>
> Urrr? You mean that if I poll the ephemeris from the chipset it'll have
> the UTC parameters? That'd be nice if true.

subframe 4 page 18 word 9 byte 1 in the ephemeris contains the leap seconds,
details here:
	http://home-2.worldonline.nl/~samsvl/nav2eu.htm

My Garmin GPS-18 has no problem dumping that to the user.  If your GPS
can show UTC it must be keeping it somewhere.

> >  If I do not have the current ephemeris the
> > GPS can not calculate the curent location.  If I can calcualte the current
> > location then I have a current ephemeris.  If I have a current ephemeris
> > I have the leap seconds.  Ergo, if I have the current location I have
> > the leap seconds.
>
> The ephemeris is a collection of words, and so long as you have the words
> needed to calculate satellite position UTC time isn't relevant.

Agreed, never said otherwise.

> I'm fairly certain that I can have current ephemeris - enough to get a
> fix from - without having yet seen subframe 4, page 18. I say this because
> I have sat outside, cold-booted my receiver, and watched as I got a fix
> in about 40 seconds, and then waited another few minutes before the time
> was properly adjusted.
>
> Or have I completely missed the point again?

I can not comment on your particular GPS.  I have seen a lot of screwed
up code in my life.  I am saying that if your GPS if properly using the
data it has, when it has it, then you have the time and fix at the
same time.

subframe 4 page 18 contains the UTC leap second AND the ionosphere
corrections.  Your GPS may be displaying the position before the time,
but it needs the iono corrections to get a GOOD fix.  If it has
the iono corrections then it has the leap seconds.

Many GPS display guesses instead of real data.  Sometimes this is called
"coast" mode or similar.  Maybe your GPS is doing something similar?
