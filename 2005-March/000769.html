Yo Eric!

On Wed, 2 Mar 2005, Eric S. Raymond wrote:

> The Garmin driver spoils this picture a bit, but Gary is working on
> a probe function.

Slowly...  Right now the garmin_probe()  gets the tty so confused
that gpsd can not recover from the damage. Even after multiple restarts
w/o the probe.  Only repeated use of stty can unfreeze the port.

Looks like I need to add some check that a garmin and/or garmin driver
is atached to the tty before sending data out the tty.

> Now, there is one big piece of configuration left.  We have to specify
> a serial device for gpsd to look at.  Our next big question is: how
> do we get rid of this requirement?  If we could manage that, gpsd
> would be completely zero-configuration.

I have a real problem with things probing my serial ports unsupervised.
My hosts have a large array of things hanging off the /dev/tty's.

To name a few in my office alone:
	fax/modems
        cell phones
	PDAs
	mice
	tablets
	printers
	Linux consoles
	Router consoles
	IP Switch consoles
	MPEG Encoder consoles
	UPS
	Alarm Systems
	Stepper motors.

Many of the dumber ones implement single character commands or
strange autobauding.  No way I want to imagine some device banging
on these without prior permission.

There is also the very real case where I might have more than one
GPS hooked up at the same time.  Maybe I am transfering waypoints,
uploading maps, etc.

gpsd should open /dev/gps by default, then implement one or all of:
	1. input tty on the command line
	2. prompt for the tty
	3. read a config file.
