Pascal F. Martin <pascal.martin at cox.net>:
> My major problem with the gpsd protocol is that I have to actually
>send a query for every item I am interested in, everytime I want an
>update.

No, you don't.  Set watcher mode.  This will trigger a send from the
daemon to all clients every time a daemon poll of the device causes
any change in status.  If you are using libgps (which I recommend) the
changes, and their timestamps, will automatically be unpacked into a
gps_data_t structure for you every time you poll.

To see how this works, look at the code in xgps.c.  It's dead simple.

> In addition I would plan for a UDP variant of the protocol. This
> would make it possible to use broadcast or multicast, another
> bandwidth saver. There is no need for reliable delivery, since the
> information is repeated periodically anyway..

That's an interesting idea.

> At the very least I would like compound requests: asking for multiple informat> ion items in a single command.

You can already do that.  Try simething like this from a telnet:

pa

and see that you get back both P and A responses.  Raw mode is really
doing it the hard way.  As I said, using libgps and watcher mode is
better.
