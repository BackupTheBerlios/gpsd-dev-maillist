"Gary E. Miller" <gem at rellim.com> writes:

> I think we are gonna back itout and go with ntohs() and friends  which
> seem to be pretty universal.

ntohs and le16toh are different.  The problem is that one is about
network (BE) vs host and the other about LE vs host.  So using ntohs on
a BE machine won't do anything (correctly, but not useful for garmin),
and it will mess things up on LE.  My impression is that this code
worked before on LE machines.

It's arguably a bug that garmin uses le instead of network byte order
but that bug is now cast in firmware.
