On Fri, 19 May 2006, Robin L Darroch wrote:

> > At least one gpsd user
> > has True North units in serious production usage.
>
> OK - in that case, we need to get that user involved in this discussion.

Agreed.

> > Also of concern is how far the effects go.  This new heading addition
> > would also need to be made to libgps, cgps, xgps, the doc, etc.
>
> Done - at least as far as libgps and the doc is concerned.  cgps and xgps
> don't need any changes, since they have no field that reports heading.

But that was my point, adding half a feature is not right.  Until cpgs
and xgps can show the data it is impossible to really validate it.

> >   Then you have all the third party apps, like gpsdrive that can
> > display track but know nothing of heading.
>
> That doesn't mean we should program gpsd to lie: it means that if gpsdrive
> wants the benefit of using heading, it should have that facility added...

Yes it should have it added, but for people using gpsdrive with all fours
wheels solidly on the ground the heading and track are the same.  So
moving track to heading for them is a fatal change.

> >  We can discuss this all we want but real users with
> > real applications always seem to have their own views.
>
> Indeed. :)  All of this has been written from the point of view of "what would
> I want if I had a True North compass?"... and the fact is, since much of my
> GPS use is in the air, I'd want a rock-solid distinction between heading and
> track.

Well until you buy an etrex you are not a real user.  NASA programmers
assert that as hard as they try, half the bugs are still found in field
tests.  Integrity can cut many ways, adding code witout being able to test
it can also imply a false integity.
