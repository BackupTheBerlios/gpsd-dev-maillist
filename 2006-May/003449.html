Rob Janssen <gpsd at xs4all.nl>:
> > Seems to me that gpsd used to do that and Eric removed that code.
> 
> I know that Eric's opinion is that gpsd should not provide the "buffering"
> functionality, and that the clients should implement it themselves.
> It has been discussed before.  I'm afraid I do not agree with it.

Neither of these evaluations is quite correct.  gpsd never did complete
buffering. What I did remove, because it seemed over-elaborate, was some
code that aged returned data values individually.

I'm not necessarily opposed to having the daemon double-buffer the data
itself.  My caveats are these:

1) It has to be possible for a client to switch off buffering.

2) The buffering has to be done *right*.

The second criterion is the nasty one.  Buffering tends to introduce
all kinds of subtle bugs -- to begin with, can there be there
circumstances under which the buffer never gets flushed?  Just
figuring out where all the edge cases are is a significant headache.
