Yo Eric!

On Fri, 10 Nov 2006, Eric S. Raymond wrote:

> Gary E. Miller <gem at rellim.com>:
> > > They're on the website.  Or you can install xmlto.  I think this
> > > reduces the problem to one we can legitimately define as "user's option".
> >
> > This breaks the users valid expectation that:
> > 	 configure;make;make install
> > is all that is needed to install a package and its documentation.
>
> Your expectation, which I agree is quite valid, depends on the
> presence of a lot more tools than just xmlto/xsltproc in the
> background.  Configure scripts require the shell.  You can't compile
> the C sources without a C compiler installed.  Etcetera.

And gcc, gas, ld, etc. are all part of the "standard" toolchain. xmlto
and xsltproc are not.

> I hardly think you'd object if configure, under the appropriate conditions,
> threw up it hands and said "Ulch!  No C compiler!  I can't go on!".  I put
> it to you that this is the same case...

Name a few non-Gnome/non-XML app that depends on xmlto.  Name a frew
packages that do not ship precompiled man apges.

> The real question here is whether we're prepared to consider XML processors
> to be part of the toolset we can take for granted.  I have no problem
> with doing that.  If you do, I'm open to argument.

Neither Slackware nor Gentoo installs xmlto or xsltproc by default.  I
expect that OS X does not either.  Judging from the complaints the last
few days on IRC there are other Distros that do not either.

> Anyway, we can basically solve this problem for RPM-based systems by
> making xsltproc a build requirement.  Anybody who uses yum or smart
> or apt-get will have the dependency chased for them automatically when
> they install the source RPM.  I'll go do that now...

That handle a few OS, what about all those that do not use rpms?

Not sure what the problem with shipping the man pages is.  Just like
a lot of packages ship the lex/yacc output to save that step for the
user we should ship the man pages.
