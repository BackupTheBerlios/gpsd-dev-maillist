> > Consider this bug squashed...
> >
> > # ./gpspipe -r 127.0.0.1 | head -10 | grep PGRME | head -1 ; echo E |
> > nc 127.0.0.1 2947
> > $PGRME,0.00,M,0.00,M,2.68,M*12
>
> There's a problem.  This makes GPSD E use 0.0 as a "no valid data" marker, a
> practice I specifically want to avoid because for some fields 0 is an
> in-band value (altitude is the most obvious example).
>
> Those zeroes in the E report should be '?' instead.  Yes, I know the
> documentation says otherwise; that's a documentation bug.  And in this
> case, backward-compatibility issues aren't because reporting of this
> field wasn't right in the first place.

OK, I'll revisit that.

I guess I need to make the PGRME generator output empty fields if the
data is not valid, to be similar to the E report. I'll also need to
adjust the parsing of the E report to try handle each error component
separately - just because we don't have all three error estimates
doesn't mean we shouldn't report what we do have.
