Marr <marr99 at highstream.net>:
> Yeah, I know that nobody will bother reading this -- they'll just
> jump to the screenshots. :^)

*I* read it.  Carefully.  Seemed to be required by the sort of, er,
management job I've fallen into.

> To me, this new version, which mimics another of my long-used GPS
> applications, provides a greater data density on the satellite
> display without sacrificing too much space. In fact, this greater
> data density is the reason that I moved the "Satellite Info" table
> to a (secondary) notebook tab (more on that below), since I don't
> see much use for it now except to provide (1) the exact satellite
> azim/elev and (2) a nice tabulated format for times when that
> display presentation is more useful (e.g. ordered by PRN).

I say well done.  The satellite picture is cool and geeky, but the 
tabular sat data never seemed very very interesting other than as
a sort of "See?  We can do this!" demonstration.  

> Like the old app, the new app displays and updates all of the same
> data coming from the GPS receiver via 'gpsd', including the new
> field to show the GPS receiver type/firmware. All that data is
> displayed in a single "tab" labelled "GPS Info" of a 2-tab
> "notebook" widget.

That's all good.

> I've taken the liberty of renaming "GPS Type" to "Receiver Type",
> since I find that term more precise. More importantly, I've renamed
> the erroneously-named "Course" to "Track", which is what the GPS
> receiver is actually providing, of course (no pun intended!).

Um, I guess I should fix that in xgps, then. Jeff, take a  note for
cgps.

> Unlike the old Motif app, which hard-codes a fixed number of
> satellites to be displayed, thereby wasting screen real-estate most
> times, this "Satellite Info" table in the new app is not constrained
> by a fixed number of satellites. The table is dynamic and resizes,
> using a vertical scrollbar as needed (which is not obvious in my
> forthcoming screenshot).

Excellent!
 
> One of the things that I always hated about the Motif version of 'xgps' is 
> that it only shows 1 line of raw incoming GPSD data and that there's no way 
> to pause the display refresh (so that you have slightly more than a 
> snowbaIl's chance in hell of reading the doggone thing).
> 
> My new version fixes both of those shortcomings.
> 
> I've added a scrollable, multi-line text area to show the raw incoming GPSD 
> data.

Again excellent.

> I've worked to keep the overall default initial application size
> under 800x600 pixels.

And a heartfelt thank you for writing a test client that doesn't require
the latest snazzy whizzy hardware to run.  

This indeed sounds very promising.
