On Sat, 11 Nov 2006, Eric S. Raymond wrote:

> Gary E. Miller <gem at rellim.com>:
> > OS X has no build system.  Many people use "fink" for that.  Dunno if
> > there is a fink package for gpsd.  The OS X users I hear about are building
> > from SVN.
>
> The fink page says: "Fink uses Debian tools like dpkg and apt-get to
> provide powerful binary package management." So the right thing is
> to make sure xsltproc/libxslt is declared as a build-time requirement
> of gpsd in the .deb the fink people are using.

And how many OS X users know how to use Fink?  Very few in my experience.

Why is it you feel the need to maintain a zillion packages instead of just
one good tar?  What possibly could be the win here?

> > That ebuild is still stuck at 2.32 for amd64 so no use to me.
>
> How's that?  I don't see any version number mentioned in it.

http://www.gentoo-portage.com/sci-geosciences/gpsd

2.32 is current for amd64.

> > > That's what package build dependencies are for.  I think the real bug
> > > here isn't failing to ship man pages, it's failing to declare the
> > > external dependency on xsltproc/xmlto properly.
> >
> > Then you have to maintain all the packages instead of just one tar.gz.
> > So many distros bork the packages so badly that  whenever I care about a
> > package I get it  straight from the maintainer.
>
> You're right.  That's why I maintain our RPM specfile and export it
> with the source distro, why I cheerfuly pulled the ebuild into the
> repo, and why I'd put a .deb template in the repo as well in a second
> if somebody offered one.  This reduces the window in which the
> packagers can screw up -- if they just take the package declaration we
> ship, everything is usually fine.

What about the other 100+++ Linux distros?  Why this need to go to all
this extra work when one simple automated fix solves (and had solved)
this for everyone?  Why work so hard to avoid the current, and simple,
standard practice?  All you need to do is look at the number of user
complaints on this before and after the change to see this violates
common user expectations and creates a big continuing support issue.
