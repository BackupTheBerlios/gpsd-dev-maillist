Hi,

Martin Schaller:
> I see only one disadvantage:
> Remote connections will be no longer possible.
> Ok, theoretically it is possible to use remote connections, but I guess
> it isn't well tested and might create some security issues.
> 
The assumption is (says so in the dbus-server manpage) that the system
dbus isn't exported on TCP sockets.

However, you can talk the dbus protocol on your own (server) socket.
No daemon necessary.

IMHO using the daemon is still a good idea for per-machine location
broadcast and gpsd configuration; remote connections should be limited
to only affect the mode of that particuar connection, not the global
state of gpsd.

> Second thing I was about to say that libgps is more lightweight than
> dbus, but at least from the shared library size this isn't true

It's also somewhat irrelevant IMHO, because I'd assume that libdbus is
already loaded on any nontrivial system anyway, these days. Thus a fair
comparison would only consider libdbus' data section.
