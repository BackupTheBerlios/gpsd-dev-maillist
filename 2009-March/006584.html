Hi,

On Do, 2009-03-19 at 11:13 +0100, Matthias Urlichs wrote:
 
> I might be opening a can of worms here, but why not use the dbus
> protocol as a base for GPSD-NG?
> 
> Advantages:
> * no more expensive parsing of float numbers etc.
> * well-defined request/response behavior
> * dbus already has a security/policy model
>   (if we use the dbus daemon)
> * ability to monitor the protocol without coding anything
>   (if ...)
> * all other things being equal, if we link to libdbus anyway, we might
>   as well use it -- instead of spending a lot of effort fixing our own
>   protocol bugs :-P

* No more changes in the libgps so version, at least for applications
which use dbus directly.
For others it might be possible to write a dbus compatible libgps.

I see only one disadvantage:
Remote connections will be no longer possible.
Ok, theoretically it is possible to use remote connections, but I guess
it isn't well tested and might create some security issues.

Second thing I was about to say that libgps is more lightweight than
dbus, but at least from the shared library size this isn't true: The
dbus library is smaller than the gps library (both stripped), which is
probably caused by linking in all gpsd stuff into libgps.

Best regards,

Martin Schaller
