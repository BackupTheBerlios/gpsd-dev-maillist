Remco Treffkorn <remco at rvt.com>:
> When I originally thought about a protocol for gpsd I had a few
> requirements.  I wanted it human readable for easy
> debugging/documentation.  It had to be easy to parse, so that
> resource constrained systems could cheaply adopt it.

I agree with these requirements.  The only way they have changed is that 
"constrained" is not as constrained as it used to be, so we could accept 
slightly more parser overhead than we used to.  

Not that I plan to; my sketch design is slightly more verbose than the
original (because we need the command-identifier space to be larger) but
isn't actually significantly more complex.

> As long as every protocol level has another in-protocol mechanism to re-
> negotiate the protocol, you can never paint yourself into a corner.
> 
> Doing it this way requires every client to support every protocol <= the one 
> it proposes, or else there would be client/server combinations that are 
> incompatible.
> 
> Since the total number of protocols for gpsd is small (==2) that
> should not be a problem?

My plan awas to distinguish NG commands with an introducer character, 
probably '?', that changes the parser's behavior for the rest of the 
current line.

So: "YO\r\n" requests a satellite report followed by a full location 
report.  But you could also do this as "Y?SKY\r\n", where the ? tells 
the parser that the rest of the line is the single NG command "SKY".
