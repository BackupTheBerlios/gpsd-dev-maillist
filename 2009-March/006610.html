There are several ways to configure a programme:

1. Compiletime - there could be a configure option that determines
the hardcoded default and the command lines available (and visible
in the help message).

2. Configuration file - n/a as yet (AFAIK?)

3. Environment variable - same as above.

4. Command line option - as I am biased towards the mobile applications
I think the traditional gpsd behavior is the best one because...
it fits my needs and it's traditional.

However, as I wrote before
gpsd opens the device at startup, even when in "wait" mode, to detect
the receiver. IMHO this should be delayed until the first client
connects. There could be a configuration file that fully describes
the unit and makes the detection unnecessary. There might be a tool,
using the same detection code, to build such a configuration file.

The delayed opening is especially usefull in portable applications
where gpsd is to be just a thin layer between the receiver and the
software that actually uses it. Think about running gpsd on a pda
with a "hot-plugged" BT GPS receiver. The hot-pluggability is for
me even more important than the power issues (i'll mesure it,
probably tomorrow).
