Eric S. Raymond pisze:

> This isn't a complete solution, because most users will never start up
> gpsd themselves.  It wiill be started for them by a hotp;ug script - so
> the question is really what mode the script should start it in.

Please remember, that there is no HOTPLUG (AFAIK) for BT RFCOMM
devices. They are created as /dev/rfcomN and are ready for use
no matter the real device is available or not. When you open it
the kernel starts trying to connect to the device.

There is nothing like polling and HOTPLUG announcement of
proximate devices. That is why I suggested some means to
preconfigure thd device in gpsd to avoid the detection phase.

I'll measure the power consumption today.

There is one more option for BT devices. You could use the bluez
library and, doing what rfcomm(1) does, connect to the receiver
"on your own". When gpsd knows that the device *is* BT it can
handle the connection timeout issues Matthias Urlichs has written
about (<20090320154729.GL23656 at kiste.smurf.noris.de>). I've noticed
them too. killing gpsd -HUP helps.
