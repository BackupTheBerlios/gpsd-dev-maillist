Chris Kuethe wrote:
> On Fri, Jul 3, 2009 at 8:43 PM, <mrdvt92 at yahoo.com> wrote:
>>> Here's my proposal for a GPSD URI format:
>>> gpsd://<host>[:<port>]/[<device>][/<protocol>]

> basically "raw", "nmea", or "gpsd" is a switch to say whether we send
> "R=2", "R=1" or "w=1"
> 
>> Something like this would be better
>>
>> gpsd://localhost:2222/dev/ttyS0?raw

   I agree that the use of question mark (or semicolon) would be better
than slash as I feel this follows more closely the spirit of RFC2396,
although admittedly the '/' version would still be RFC2396 compliant.


   However, rather than restricting the query parameter to one
hard-coded use, how about using it simply to pass arbitrary commands to
GPSD.  E.g.,:  gpsd://localhost:2947/dev/ttyS0?R or gpsd://localhost?R=1
would achieve the same effect as above.  As a usability exception, one
could make an empty query equivalent to "?W=1", and one accepts that
this proposal introduces some redundancy (and conflict) with the path
part of the URI (i.e., gpsd://localhost/dev/ttyS0 vs
gpsd://localhost?F=/dev/ttyS0) but I cannot identify any major issues
which would be caused by that.

   So, my suggestion is:

gpsd://<host>[:<port>]/[<device>][?<commands>]

   or

gpsd://<host>[:<port>]/[<device>][;<commands>]

>> I assume that we would also support
>> gpsds://login:passwd@server:port/dev?protocol
> 
> maybe. are you planning on making the stream encrypted or just using a
> plaintext user/pass? where will login information be stored?

   Perhaps it would make sense for the URI parser to support (and for
now ignore) the authentication parts of the URI until such time when a
"secured" GPSD comes around?
