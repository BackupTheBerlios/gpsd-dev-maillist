Chris Kuethe wrote:
> This came to me while I was answering the raw data thread.
> 
> Someone was asking me why I'd want to make gpsd be able to use gpsd as
> a source device... now I have an example where this would be useful. I
> can have a long-running gpsd watching a receiver, and then I can run
> "gpsd -S 2948 gpsd://127.0.0.1:2947/raw" and be hacking on other
> things. Or use one gpsd as an aggregator for a bunch of remote
> servers.

I'm wondering if it would make sense to allow gpsd to work as backend for
gpsd-proxy (http://gpsdproxy.sourceforge.net/) - it uses udp to transport gpsd
data, for example from embedded devices. At the moment there is only some Python
code in the gpsdproxy project to write the receved packets to a psql db, but it
would probably be useful either to include the proxy functionality completely
(so you don't need an extra gpsdproxy proxess), or to support reading the udp
packets.

Just an idea, I don't have a need for such a proxy....
