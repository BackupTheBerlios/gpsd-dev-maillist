If it helps, given that my patch uses quite different handling at the 
back-end, you could try applying that instead.  It's written with the 
numbers the other way around (J=1 is jitter ON i.e. 
clear-at-start-of-cycle), but otherwise it works the same way.

It doesn't currently have a wired-policy switch (although it *does* 
offer a choice of default buffering logic per-client at build time),

don't change (at least, not by more than one boolean value) based on 
the number of clients, the impact on CPU & memory load would be even 
less significant.

Up to you, but since we both finished quite different patches to 
accomplish the same task at the same time, it seems like the logical 
option to try putting it in and seeing if it will handle the
