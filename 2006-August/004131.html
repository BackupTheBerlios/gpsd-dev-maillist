On 8/17/06, Chris Kuethe <chris.kuethe at gmail.com> wrote:
> how do we want to handle single-protocol builds, especially regarding
> client tools? if i say --disable-sirfII, should i still get a fully
> functional sirfmon/sirfflash, should i get a stub program that says
> "this feature was not compiled in", or should i not get these tools at
> all?

Since the client tools could be run (and perhaps built) on a totally
different machine, I think that sirfmon should be able to connect to a
gpsd that doesn't support sirf and be told that it's not supported.

For example, I build and run gpsd on my gumstix, but I build and run
the client tools on my Fedora box (because it has the nice GUI).

> also, it doesn't make much sense to build a gpsd with all protocols
> disabled. should we detect that and throw an error?

Seems reasonable. I don't think it matters if its a compile time check
or a runtime check, but it seems like a reasonable thing to detect.
