Remco Treffkorn <remco at rvt.com> writes:

> On Monday 24 May 2010, Greg Troxel wrote:
>> 
>> Remco Treffkorn <remco at rvt.com> writes:
>> 
>> > That is because we are setting the tty to raw mode, which is good for
>> > arbitrary binary protocols. In case of NMEA we could just set the tty
>> > line mode (cooked, but with most of the processing (including echo)
>> > turned off).
>> 
>> I think the problem is not with real serial ports.
>
> The usb dongle stuff gets still processed by the tty layer. There is
> no difference for the upper layers. Real serial ports can have the
> same problem, it is just less likely.

Agreed.  What I meant is that I think all the cases of this bug actually
causing problems have been with other than real serial ports.  With a
real serial port, the likelihood of delays long enough to get multiple
sentences in a read is small, and then each new character triggers a
read which might process a sentence from the buffer.  So I don't think
it's possible for persistent trouble to arise with a real serial port.

> In line mode a read is guaranteed to return a string in the buffer
> that contains a newline (ie cr/lf) exactly once and as the last two
> bytes of this string, even if there is more data in the tty input
> buffer. This means you always only get exactly one sentence.

Sure, but gpsd doesn't know what kind of device is present, so it
doesn't make sense to be in other than raw mode.
