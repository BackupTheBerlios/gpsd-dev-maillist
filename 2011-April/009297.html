You can certainly measure two one-way trip times.  The problem is that
NTP is using the same measurements to steer the clock, so persistent
asymmetry will result in a clock offset.  Absent GPS/etc., it's not
possible to disambiguate these.

With GPS, I don't think you need to get into shared memory interfaces.
ntpd steers the clock, and so the NTP timestamps are in the local
(synchronized) clock and you can just subtract them.  The differences
between system time and GPS observations should become very small after
running for a while, and if not there's a synchronization problem to be
solved.   It migth be useful for gpsd to log the differences between
system time and observed GPS-derived time and plot that over time.

An important point, I think, for this usage is that delays are usually
low and then buffers fill up during transfers.   So even with NTP being
steered from remote observations, the onset of increased one-way delays
will be readily observable.
