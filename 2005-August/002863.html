On 8/12/05, Wolfgang S. Rupprecht
<wolfgang+gnus20050812 at dailyplanet.dontspam.wsrcc.com> wrote:
> 
> What is the current feeling about gpsd and gps initialization? I now
> have the situation that a home-grown program I run turns off all the
> sirf msgs it has no need for as soon as the gps send it one of them.
> Among other things, it turns off msg2 which is the x-y-z version of
> the navigation msg.  This is the navigation msg that gpsd uses (and
> needs).  Unless I run a fix-up program before running gpsd, gpsd just
> sits there answering "p=?" to any position requests.  Should gpsd ask
> for the msgs it needs to do its job, or is the feeling that it would
> be rude for a program to silently reprogram a user's gps on them?

This is only slightly different from gpsd putting the receiver into
binary mode. In the case of gpsd going away suddenly you could be left
with a receiver in binary mode. I'd be ok with gpsd configuring the
hardware to get the information it needs. Document it of course, but
isn't the purpose of a hardware driver to talk to the hardware, and do
stuff with the end result of making the hardware useful to the user?

Put another way: your tool configures your receiver so that it can
best do the job you're asking of it, why can't gpsd do the same?
