>> The different drivers use varying techniques to "pack" the data for the

> Yes, the same problem presents itself with Garmin binary mode.  Since
> sat can only be numbered 1 to 32 this makes a lot of sense to me.  It
> would simplify a lot of things.  We should seriously consider for 2.30.

I don't think it should be indexed by PRN.  That could cause problems when
fake PRNs are used for special purposes like WAAS.
The index I propose is the receiver channel number (starting from 0),
where each element holds PRN, azimuth, elevation and signal strength.
Zeroes are stored when a channel is not in use.
The "gpsdata.satellites" variable should be renamed "gpsdata.channels" and
indicates how many members of the arrays can be actually in use for this
specific receiver (value like 12).
