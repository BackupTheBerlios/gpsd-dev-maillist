> Does anybody else remember that the original NCSA Mosaic had this bug?
> Whenever the download animation ran, it would take 100% cpu.
>
> They depended on the fact that on some systems the timeout value in
> select()
> does not get clobbered.

If I remember correctly, it was a Linux innovation to return the remaining
timeout, so there were quite some ported programs that had problems in
this area.

select() always has been a bit odd.  It is not a general solution for the
"this process needs to wait for the next interesting event" problem.
We see this in gpsd, where we would like to wait for a DCD transition but
it cannot be done with select(), and we need to start a separate thread
for it.

Not sure what is better...  do everything with threads and have all the
threads block in a read or similar, or have a very general system call
that can return on any condition that you think of.
