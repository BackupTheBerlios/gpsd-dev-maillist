On Tue, Oct 26, 2004 at 07:27:03PM -0400, Eric S. Raymond wrote:
> Is there any real win over select(2) here?

Sorry, I had replied before I saw your followup question.

No, I don't think there is a real win.

GLib encapsulates select(2) in a manner that allows your code to be
portable to all systems on which GLib has been ported, which is
allegedly a fair few, but I haven't memorised the full list; I'm only
interested in Linux.  

GLib is altogether different to glibc.  It is maintained by the GTK+
team.

GLib is a portability wrapper at code level, and you're already using
autoconf as a portability wrapper at source #include/#ifdef level.

It might be beneficial to use GLib's I/O channels if there were plans
for significant changes to the event model, since GLib abstracts
functionality in a way that makes coding event handling programs
slightly easier.

So I can certainly see why the author of ngpsd used it.
