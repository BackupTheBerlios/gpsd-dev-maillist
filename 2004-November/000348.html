Eric S. Raymond wrote:
> James Cameron <quozl at us.netrek.org>:
> > I had a quick look at the main loop, and the only thing odd I saw was
> > that a SIGALRM handler calls a function that calls Motif libraries; and
> > I thought these were not signal reentrant.  At least I was fairly sure X
> > library isn't.
>
> Yes, which is why I suspend signals unyil that call reyurns.

Unfortunately that is not sufficient.  The functions for the widget
might be active on the stack already at the time the SIGALRM is
delivered; e.g. as a result of calling update_display() from the X input
handler via libgps, or if there is any event such as expose, key or
mouse press.

("man XtNoticeSignal" says it's the only Intrinsics function that can
safely be called from a signal handler.)

Instead, use X toolkit timeouts.  I've attached a patch that changes
gps.c from using SIGALRM to using XtAppAddTimeOut. 

I also changed the logic slightly so that the duration of the current
status is calculated from the system time, rather than from the number
of alarm calls.

I've tested it with my FV18.  It worked fine, and also handled reporting
the loss of data if gpsd was sent a SIGSTOP, or if the cable to the GPS
was pulled out.


Also, I hacked into the widget creation code, simplifying it and
removing what seemed to be unnecessary widget arguments.  I've tested
this too, and it is more reliable at responding to window resize
actions.

- changed parentage to make a left and right half, children of the form,
- removed form attachment arguments from widgets that weren't in a form,
- removed the fallback resources in favour of setting the labels when
  the widgets are created,
- set a new default geometry after the changes.

I'd like someone to test my gpsd-geometry.patch please.
