Eric Raymond wrote:
> Bernd Zeimetz <bernd at bzed.de>:
>> - Add a script which pushes to github and similar pages regulary, thats an
>> easy and cheap backup (and safe due to the checksums git uses) and it
>> also allows other peopel to clone the codebase easily to add their
>> changes and ask to pull them.
> 
> What would trigger the script?  You want one of us to run it by hand
> occasionally, or do you want it in a commit hook?

I'd just use a cronjob.

>> - Provide "stable" branches, similar to what the kernel developers
>> do. For distributions like Debian it is important to be able to
>> update packages in case of important bugs easily. And it makes
>> things much more easy for packagers if such bugfixes are available
>> in the upstream's repository. Otherwise you have to find the right
>> commits, pick them onto the older code base
> 
> I think master *is* the stable branch.  If we need to create an
> unstable later, fine, but I don''t think anyone has plans for
> destabizing changes in the foreseeable future.

But it is not possible to use the master branch as an update for bugs of older
revisions. Stable updates in Debian only allow to include bugfixes, no new
features. The master branch includes new features, so there is no way for me to
use it. As long as it is only me it is probably not worth to create stable
branches, but other packages might benefit from it, too.

>> - Never allow to push non-fast-forwards. Otherwise it will be a PITA
>> to use for others.
> 
> Explain?

You might want to have a look at
http://eagain.net/articles/git-for-computer-scientists/ to understand the theory
behind git. Non-fast-forward pushes drop existing parts from the graph.
