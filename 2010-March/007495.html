Miroslav Lichvar <mlichvar at redhat.com> writes:

> On Wed, Mar 24, 2010 at 01:39:43AM -0400, Eric Raymond wrote:
>> > With all of the changes, can we change the naming convention.  In
>> > RPM speak version 2.93 is less than 2.93dev which is not what gpsd
>> > intends.
>> 
>> I'm open to suggestions.  I think the "dev" suffix is a BSD convention
>> imported by ckuethe.  What's the correspomding practive in Fedoraland?
>
> Not sure if rpm supports the tilde sorting feature, but the official
> Fedora guidelines say the suffix should be added to the release tag
> instead of version. Something like gpsd-2.93-0.1.dev.
>
> http://fedoraproject.org/wiki/Packaging:NamingGuidelines#NonNumericRelease

AFAIK dev is not a BSD convention.  It might be from openbsd ports.

My view is that version numbers should sort correctly with the minimum
number of special-case rules.  In pkgsrc, the following rules exist:

 const test_t   modifiers[] = {
        {       "alpha",        5,      Alpha   },
        {       "beta",         4,      Beta    },
        {       "pre",          3,      RC      },
        {       "rc",           2,      RC      },
        {       "pl",           2,      Dot     },
        {       "_",            1,      Dot     },
        {       ".",            1,      Dot     },
        {       NULL,           0,      0       }
};

This handles 2.93.1 which is a patch release from 2.93 (gpsd doesn't
seem to issue those, which is fine).  2.94a2 is an alpha which sorts
before 2.94, so

2.94a1 < 2.94b1 < 2.94rc3 < 2.94 < 2.94.1

But, this is all more complicated than it needs to be because people
have not used the long ago standard FSF practice of using .80 for alpha
and .90 for beta.  So you'd have 2.93.80 now, and 2.93.94 for the fifth
beta on the way to 2.94.  This always seemed highly sensible.

So I would just bump the version in git to 2.93.80 and leave it there.
Packaging systems should be discouraged from packaging intermediate
versions - and therefore whenever someone wants to package from git
that should be a hint that there should be a release.
