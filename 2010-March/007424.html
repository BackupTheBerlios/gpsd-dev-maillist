Eric Raymond <esr at thyrsus.com> writes:

> Greg Troxel <gdt at ir.bbn.com>:
>> I wonder if having a score for each matcher, and keeping an EWMA would
>> help.  then false alarms would be below a steady lock.
>
> EWMA?  Explain, please.

exponentially weighted moving average.  Each epoch, estimate probability
of match, and then set

avg = avg * 7/8 + estimate * 1/8

This is used in TCP round-trip-time estimation.  The idea is that if a
decoded was always saying 'good lock', the value would stay close to
1.0.  A decoded that occasionally saw a lock but usually didn't would be
low, and thus not cause a switch.  But a lock when everything else is
not hitting would still win.  Maybe that's too complicated.

> Note as you think about this that TSIP kind of breaks the
> autoconfiguration model.  All the other packet types are checksummed,
> which makes false positives wildly unlikely.  So I think giving gpsd
> the ability to recover from a false positive lock on TSIP should
> suffice.

So what I'm saying is that lock on TSIP shouldn't count as a lock if
another decoded has a lock, and unless it's consistent over enough bytes.
