I suggested branches, and Eric pointed out that because we have
regression tests we are in little danger of shipping broken code.
That's true (and I really appreciate the tests), but the problem is that
our current procedures result in make check being run for only one
platform (typically Linux/amd64) to check a proposed change before it is
pushed to the official repository master branch.

So a proposal:

  1) Changes that are likely to have portability issues should be put on
  a branch with a call on the list for "please check this branch" and
  then a 72 hour delay before merge to master.

  2) Commits that break make check on any platform where it has been
  believed to work recently that didn't do the above must be reverted
  immediately upon posting of a complaint.  The key is to put the onus
  on not breaking things - everywhere - on the changer.

Of course, "likely to have issues" is a judgement call, and the point of
rule 2 is to sharpen the point - if rule 2 is invoked, the judgement in
point 1 was wrong.

Thoughts?
