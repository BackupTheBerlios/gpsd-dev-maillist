On Wed, Mar 31, 2010 at 17:06, Eric Raymond <esr at thyrsus.com> wrote:

> This is an attempt to round up the results of the Qt port disussion and
> list the things we need to do before we can announce the feature.
>
> 1. The Qt port needs to be part of our regular test sequence.  This means:
>
>   * We need to teach configure.ac to detect Qt/qmake.  I believe Greg
> Troxel
>     has volunteered for that.
>
>   * Set an automake conditional flag appropriately.  That should be trivial
>     for Greg to do once he's accomplished the previous.
>
>   * Use that conditional to wrap a Makefile.am production that runs qmake
>     to make the libQgpsmm library.  That sounds like Chris Browet's job
>     to do.
>
>   * Once that's done, I'll set up the regression-test productions myself.
>
> 2. Handling build-from-git and the generated-files issue.
>
> I'm accepting that we need to support Qt builds from git and it
> is thus not sufficient for the generated stuff to be in the release
> tarballs.
>
> I'm also accepting that, at least for the moment, we'll need to carry
> the special Qt config.
>
> The question I now have is whether we can assume that the build
> platform (not the run target, just the build platform) will support
> (a) Python, and (b) scons.
>
> If we can assume Python. then we simply add a step to the .pro file
> to run ais_json.py to produce ais_json.i.  I would strongly prefer
> that to having the latter in the repo.
>
> Being able to assume scons doesn't solve any immediate problem, but
> would make me consider using scons for unified cross-platform build,
> getting rid of both the Qt pro file and our autotools stuff.
> --
>                <a href="http://www.catb.org/~esr/<http://www.catb.org/%7Eesr/>">Eric
> S. Raymond</a>
>

Re 2.a, I think we can safely assume it;
Re 2.b, from the scons project page, it claims to support most of the common
platforms/language + support Qt directly apparently (incidentally, scons is
written in python, so b = a)

Re ais_json.i, it is generated by jsongen.py, isn't it? A further problem is
that jsongen.py is itself generated from jsongen.py.in at configure time,
apparently, so for non-autogen platforms, this is still a problem (although
the only thing the configure step do is changing the #! python header)
