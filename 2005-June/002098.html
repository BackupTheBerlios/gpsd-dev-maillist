Another issue about the structure/architecture of the daemon:

After sending the W command (e.g. on a "telnet localhost gpsd"), the
daemon issues several O outputs per cycle.  Some of them include more info
than others, probably caused by the different GPS messages from which they
are derived.
This is quite problematic with the TSIP driver, because in TSIP the info
is spread over so many different message types.

GPSD,O=ID5c 1118518474.00 0.005 52.062643 5.138538   10.65  0.00  0.00  
0.0000   0.000  0.000 ?      ?      ?
GPSD,O=ID84 1118518475.00 0.005 52.062643 5.138538   10.65  0.00  0.00  
0.0000   0.000  0.000 ?  0.00      ?
GPSD,O=ID6d 1118518475.00 0.005 52.062643 5.138538   10.65  0.00  0.00  
0.0000   0.000  0.000 ?      ?      ?
GPSD,O=ID82 1118518475.00 0.005 52.062643 5.138538   10.65  0.00  0.00  
0.0000   0.000  0.000 ?      ?      ?

But I also see it occuring with SiRF.

GPSD,O=MID18 1118518584.00 0.005 52.062661 5.138618   33.27 16.00  8.00
-115.3555    0.006  0.000 ?      ?      ?
GPSD,O=MID98 1118518585.00 0.005 52.062660 5.138618   33.25 16.00  8.00
-107.3776    0.004 -0.001 ? 32.00      ?

Shouldn't the daemon send only 1 aggregated O output per cycle?
Note that in the above the O=xxx field does not show the type of the
message that triggered this output, but the type of the message before
that one.
